01 · Introducción

Programación Concurrente

Fundamentos teóricos, vocabulario esencial y los cinco modelos de concurrencia estudiados en la materia.

Definiciones Clave

Programa

Conjunto de datos, asignaciones e instrucciones de control de flujo que compilan a instrucciones de máquina. Se ejecutan secuencialmente en un procesador.

Programa Concurrente

Conjunto finito de procesos secuenciales que pueden ejecutarse en paralelo. Su ejecución resulta del intercalado arbitrario de instrucciones atómicas de cada proceso.

Instrucción Atómica

Una instrucción que puede ejecutarse de principio a fin sin interrupciones, o directamente no ejecutarse. No hay estado intermedio observable.

Proceso

Cada programa secuencial que conforma el programa concurrente. Tiene su propio estado de ejecución.

Sistema Paralelo

Varios programas ejecutándose simultáneamente en procesadores físicos distintos.

Multitasking

Ejecución de múltiples procesos en un periodo de tiempo. El scheduler del kernel coordina el acceso.

Multithreading

Construcción que permite ejecución concurrente de threads dentro del mismo proceso/programa.

Desafíos de la Concurrencia

⏱
Sincronización
Coordinación temporal entre distintos procesos.
📡
Comunicación
Datos que los procesos necesitan compartir para cumplir la función del programa.
⚠️
Race Conditions
Cuando el resultado depende del orden de ejecución de los threads.
🔒
Deadlock
Dos o más procesos bloqueados mutuamente esperando recursos que el otro tiene.

02 · Lenguaje

Introducción a Rust

Rust es el lenguaje base de la materia. Sus objetivos son velocidad, seguridad y concurrencia. Su sistema de ownership permite garantizar "concurrencia sin miedo".

🚀
Velocidad
Zero-cost abstractions. Performance comparable a C/C++.
🛡
Memory Safety
Sin punteros sueltos, sin buffer overflow. Todo en tiempo de compilación.
⚙️
Concurrencia
El compilador previene race conditions. Si compila, no hay data races.
📦
Cargo
Herramienta de build + gestión de dependencias (crates.io).

Ownership: el corazón de Rust

⚠
Ownership es la razón por la que Rust puede garantizar memory safety sin garbage collector. Inspirado en C++ RAII pero llevado al extremo: el compilador verifica las reglas en compile-time, con costo de runtime cero.
1
Cada valor tiene un único dueño (owner)

Solo una variable puede ser el owner de un valor en un momento dado.

2
Cambiar de dueño es un move

Al asignar a otra variable o pasar a una función, la original ya no puede usarse.

3
Cuando el owner sale de scope, el valor se elimina

Se llama automáticamente al trait Drop. No hace falta free().

Traits Send y Sync

Send

El ownership del tipo puede ser transferido entre threads. Casi todos los tipos son Send. Excepción: Rc<T>.

Sync

Es seguro ser referenciado desde múltiples threads. T es Sync si &T es Send. Los primitivos son Sync.

Threads en Rust

rustuse std::{thread, time::Duration};

let join_handle = thread::spawn(move || {
    println!("[HIJO] spawnie");
    thread::sleep(Duration::from_secs(2));
});

join_handle.join().unwrap();  // espera que el thread termine
03 · Modelo de Paralelismo

Fork-Join

Modelo de paralelismo datado de 1963 (Melvin Conway). El cómputo se divide recursivamente en sub-tareas independientes que se ejecutan en paralelo y luego se combinan.

🔀
División Recursiva
El cómputo se parte recursivamente. Los sub-cómputos son independientes → se pueden ejecutar en paralelo.
🎲
Determinístico
Los threads están aislados. Produce el mismo resultado independientemente de la velocidad.
⚡
Performance Ideal
En caso ideal: t_secuencial / N_threads. Varía por tamaño de tareas y costo de combinar.

Work Stealing

El algoritmo de scheduling más importante en Fork-Join. Resuelve el balanceo de carga cuando las tareas tienen distintos tamaños. Cada thread tiene su propia cola de tareas; cuando una cola se vacía, el thread "roba" tareas de la cola de otro.

Implementación con Rayon

rustuse rayon::prelude::*;

// Dos tareas en paralelo
let (v1, v2) = rayon::join(fn1, fn2);

// N tareas en paralelo con iterador paralelo
giant_vector.par_iter().for_each(|value| {
    do_thing_with_value(value);
});

05 · Modelo Asincrónico

Async / Await

Modelo de concurrencia basado en la cooperación entre tareas livianas. Ideal para operaciones I/O-bound donde los threads pasarían la mayor parte del tiempo esperando system calls.

❌ No usar async

Calcular un factorial, producto de matrices, problema de los filósofos. Son tareas CPU-bound → usá Fork-Join.

✓ Usar async

Consultar servicios externos, leer archivos, servir requests HTTP. Son tareas I/O-bound donde el thread estaría bloqueado esperando.

⚠
El cambio de una tarea a otra ocurre SÓLO en las expresiones await. Un cómputo grande en una función async no daría lugar a la ejecución de otras tareas. Esto es diferente a los threads, donde el scheduler puede interrumpir en cualquier momento.
06 · Sincronización

Corrección & Locks

En programas concurrentes, la corrección es más difícil de razonar que en programas secuenciales: la salida puede depender del escenario de ejecución.

Propiedades de Corrección

Safety — debe ser verdadera siempre
Exclusión Mutua Dos procesos no deben intercalar ciertas (sub)secuencias de instrucciones.
Ausencia de Deadlock Un sistema que aún no finalizó debe poder continuar realizando su tarea.
Liveness — debe volverse verdadera eventualmente
Ausencia de Starvation Todo proceso que esté listo para utilizar un recurso debe recibirlo eventualmente.
Fairness (equidad) Una instrucción que está continuamente habilitada, eventualmente aparece en el escenario.

RwLock en Rust

rustuse std::sync::{Arc, RwLock};

let shared = Arc::new(RwLock::new(0i32));
let mut w = shared.write().unwrap();
*w += 1;
07 · Sincronización II

Semáforos

Mecanismos de sincronismo implementados como una construcción de programación concurrente de más alto nivel. Introducidos por Dijkstra en los años 60.

Semáforo

Tipo de dato compuesto por un entero no negativo V y un conjunto de procesos L (los bloqueados). Dos operaciones atómicas: wait(S) y signal(S).

Semántica de las Operaciones

wait(S) — p(S)
if S.V > 0
    S.V := S.V - 1
else
    S.L.add(p)
    p.state := blocked
signal(S) — v(S)
if S.L.isEmpty()
    S.V := S.V + 1
else
    q := S.L.takeOne()
    q.state := ready

Problema Clásico: Productor-Consumidor

CasoSemáforos necesariosDescripción
Buffer infinito notEmpty (0, ∅) Solo problema 1: no consumir si el buffer está vacío
Buffer acotado N notEmpty (0, ∅)
notFull (N, ∅)		 Problema 1 y 2: además no producir si el buffer está lleno
08 · Sincronización III

Monitores

Herramienta de sincronización de más alto nivel que los semáforos. Los monitores encapsulan estado compartido junto con las operaciones que lo modifican, garantizando exclusión mutua implícita.

Monitor

Herramienta que permite a los hilos tener exclusión mutua y la posibilidad de esperar (block) a que una condición se cumpla. Tiene un mecanismo para señalizar a otros hilos cuando su condición se satisface.

Condition Variables

OperaciónEfecto
waitC(cond) Bloquea el proceso y libera el lock del monitor. Siempre bloquea.
signalC(cond) Desbloquea el proceso del tope de la cola FIFO. No tiene efecto si la cola está vacía.
empty(cond) Devuelve true si la cola de la condition variable está vacía.

Condition Variables en Rust

rustuse std::sync::{Arc, Condvar, Mutex};

let pair = Arc::new((Mutex::new(false), Condvar::new()));

// Thread que espera:
while !*ready {
    ready = cvar.wait(ready).unwrap(); // libera el lock y espera
}

// Thread que señaliza:
*ready = true;
cvar.notify_all();
⚠
Spurious wakeups: Un thread puede despertar de un wait sin que notify haya sido llamado. Por eso siempre se debe volver a verificar la condición en un while, no en un if.
Clásico
Filósofos, Barbero, Lector-Escritor
▼

Filósofos: 5 filósofos, 5 palitos. Si todos toman el palito izquierdo → deadlock. Solución: árbitro o forzar a uno a tomar en orden inverso.

Barbero Dormilón: Si no hay clientes, el barbero duerme. El primer cliente lo despierta. Sincronizar despertar con llegada de clientes.

Lector-Escritor: Múltiples lectores simultáneos OK. Escritor requiere acceso exclusivo. Cuidado con starvation de escritores.

09 · Formalismo

Redes de Petri

Formalismo matemático para modelar sistemas concurrentes. Permite razonar sobre propiedades como deadlock, alcanzabilidad y corrección de manera visual y rigurosa.

●
Lugar (p): representa un estado o condición
Transición (t): representa un evento o acción
Token (•): indica que un lugar está marcado / activo

Red Ordinaria de Petri

Definición Formal

PN = (T, P, A) donde T = transiciones, P = lugares, A ⊆ (T × P) ∪ (P × T) = arcos. Una transición t está habilitada si M(p) ≥ 1 para todo p ∈ I(t). Al disparar: consume tokens de entrada, produce tokens de salida.

Red General de Petri (con pesos)

Definición Extendida

PN = (T, P, A, W, M₀) donde W : A → ℕ es la función de peso. Una transición t está habilitada si M(p) ≥ W(p, t) para todo p ∈ I(t).

10 · Redes de Petri · Avanzado

Red General & Ejemplos Clásicos

Modelo Petri
Mutex (Exclusión Mutua)
▼

P2 inicializado con 1 token actúa como el lock. Solo uno de los procesos puede tener ese token y entrar a la sección crítica. P3 y P4 nunca tienen token simultáneamente → exclusión mutua garantizada.

Modelo Petri
Productor-Consumidor con Buffer Acotado (N)
▼

Se agrega lugar p₆ con N tokens que representa los espacios libres. La transición que produce necesita un token de p₆ (hay espacio). La que consume devuelve un token a p₆. El productor no puede superar N items.

Modelo Petri
Lector-Escritor con Arco Inhibidor
▼

Arco Inhibidor: Se representa con flecha de punta abierta. Una transición con arco inhibidor desde p solo está habilitada si p NO tiene tokens. T0 (lector) tiene arco inhibidor desde P4 (escritores activos). T2 (escritor) tiene arco inhibidor desde P1 (lectores activos).

11 · Pasaje de Mensajes

Channels & Mensajes

Modelo de comunicación basado en el pasaje de mensajes entre procesos. En lugar de memoria compartida, los procesos se comunican enviando datos a través de canales tipados.

💬
Filosofía Go adoptada por Rust: "Do not communicate by sharing memory; instead, share memory by communicating."

Modelos de Comunicación

Comunicación Sincrónica

El emisor se bloquea hasta que el receptor recibe el mensaje. No hay buffer.

Comunicación Asincrónica

El emisor deposita el mensaje en un buffer y continúa. El receptor lo lee cuando puede.

Direccionamiento Simétrico

El emisor nombra al receptor y el receptor nombra al emisor.

Flujo Unidireccional

Los canales solo transportan datos en una dirección: de emisor a receptor.

Canales en Rust — std::sync::mpsc

Rust provee canales MPSC (Multiple Producer, Single Consumer). El canal tiene dos extremos: un emisor Sender<T> y un receptor Receiver<T>. Transfieren el ownership del elemento enviado.

rustuse std::sync::mpsc;
use std::thread;

let (tx, rx) = mpsc::channel();

thread::spawn(move || {
    tx.send(String::from("Hola")).unwrap();
});

let received = rx.recv().unwrap();
println!("Recibido: {}", received);

Selective Input

Selective Input

Sintaxis que permite escuchar en varios canales de forma bloqueante y desbloquearse con el primero que recibe un mensaje. Fundamental para que un proceso pueda reaccionar a distintas fuentes de datos.

Canales en Unix: Pipes, FIFOs y Message Queues

Unix provee Pipes y FIFOs para conectar dos procesos independientes, orientados a bytes. Los FIFOs tienen representación en el filesystem. También existen Message Queues, orientadas a mensajes como unidades independientes.

Filósofos con Canales — Solución de Chandy/Misra

✓
Idea clave: Los palitos circulan por mensajes entre filósofos. Cada palito puede estar sucio o limpio. Un filósofo entrega un palito sucio si le es solicitado, pero retiene uno limpio. Esto garantiza ausencia de deadlock y starvation.
12 · Modelo de Actores

Actores & Actix

Modelo de concurrencia desarrollado por Carl Hewitt en 1973. Popularizado por Erlang. Los actores encapsulan estado, comportamiento y tienen una dirección única.

📬
Mailbox (FIFO)
Cada actor tiene una cola de mensajes entrantes. Procesa uno a la vez.
🔒
Estado Privado
No comparte memoria con otros actores. El estado solo cambia al procesar mensajes.
🌐
Distribuible
La dirección de un actor puede ser remota. El código no cambia.
⚡
Livianos
Se pueden crear miles de actores en lugar de threads.

Formas de Enviar Mensajes (Actix)

MétodoComportamiento
Addr::do_send(M) Fire-and-forget. Ignora errores. No retorna resultado.
Addr::try_send(M) Envío inmediato. Retorna SendError si mailbox lleno o cerrado.
Addr::send(M) Retorna un Future con el resultado del handler. Permite .await.
13 · Sistemas Distribuidos

Concurrencia Distribuida

En sistemas distribuidos, los procesos corren en máquinas separadas y se comunican por red. Los algoritmos clásicos de exclusión mutua y comunicación deben adaptarse a este modelo.

Exclusión Mutua Distribuida

Centralizado
  • Un proceso es el coordinador
  • Para entrar a la SC: enviar mensaje al coordinador
  • Si SC libre: coordinador responde OK
  • Si SC ocupada: coordinador encola y espera

⚠ Punto único de falla

Ricart-Agrawala
  • Enviar REQUEST con timestamp a todos
  • Recibir OK de todos para entrar
  • Timestamps desempatan conflictos
Token Ring
  • Procesos en anillo lógico
  • Solo el que tiene el token puede entrar
  • Al salir, pasa el token al siguiente

Flujo TCP: Cliente y Servidor

🖥 Servidor TCP
socket() — crear socket
bind(addr) — asignar dirección local
listen(backlog) — poner en escucha
accept() — esperar conexión entrante
read(buf) — leer datos
write(buf) — enviar respuesta
close() — cerrar conexión
💻 Cliente TCP
socket() — crear socket
— — —
— — —
connect(addr) — conectarse
write(buf) — enviar petición
read(buf) — leer respuesta
close() — cerrar conexión
14 · Distribuido II

Algoritmos de Elección

Varios algoritmos distribuidos requieren un coordinador con un rol especial. Los algoritmos de elección permiten elegir uno automáticamente cuando el actual falla.

Bully (Garcia-Molina)

Cuando el coordinador no responde, P envía ELECTION a los de mayor ID. Si nadie responde, P es el coordinador. El mayor ID siempre gana. O(N²) mensajes en el peor caso.

Ring

Procesos en anillo lógico. El mensaje ELECTION recorre el anillo acumulando IDs. Al completar la vuelta, el mayor ID gana. O(N) mensajes.

Comparación
  • Bully: simple, re-elección automática
  • Ring: eficiente en anillos
  • Token Ring: equitativo, sin starvation
15 · Distribuido III

Transacciones Distribuidas

Una transacción es una secuencia de operaciones que debe ejecutarse de forma atómica en un sistema distribuido. Si cualquier parte falla, todo debe revertirse.

Propiedades ACID

A
Atomic
La transacción no puede ser dividida. Todo o nada.
C
Consistent
Cumple con todos los invariantes del sistema.
I
Isolated
Las transacciones concurrentes no interfieren entre sí.
D
Durable
Una vez commiteados los cambios, son permanentes.

Primitivas de Transacción

PrimitivaDescripción
BEGIN TRANSACTIONMarca el inicio de la transacción
END TRANSACTIONFinalizar la transacción y tratar de hacer commit
ABORT TRANSACTIONFinalizar forzadamente y restaurar los valores anteriores (rollback)
READ / WRITELeer o escribir datos de un archivo u otro objeto

Commit en Dos Fases (2PC)

El protocolo más utilizado para garantizar atomicidad en transacciones distribuidas. Un coordinador coordina a los participantes:

⬡ Fase 1 — Prepare
1
El coordinador escribe prepare en su log y envía mensaje PREPARE a todos los participantes
2
Los participantes que pueden commitear responden READY y escriben en su log
3
Si alguno responde ABORT, la transacción se cancela
⬡ Fase 2 — Commit
1
Si todos respondieron READY: el coordinador escribe COMMIT y envía mensaje COMMIT
2
Los participantes escriben COMMIT en el log y responden FINISHED
3
El coordinador escribe END OF TRANSACTION

Implementaciones

Técnica
Private Workspace
▼

Al iniciar una transacción, el proceso recibe una copia de todos los archivos a los que tiene acceso. Hasta que hace commit, trabaja con la copia. Al commitear, se persisten los cambios. Desventaja: extremadamente costoso salvo por optimizaciones.

Técnica
Writeahead Log (WAL)
▼

Los archivos se modifican in place, pero se mantiene una lista de los cambios aplicados (primero se escribe la lista y luego se modifica el archivo). Al commitear, se escribe un registro commit en el log. Si la transacción se aborta, se lee el log de atrás hacia adelante para hacer rollback.

Control de Concurrencia
Two-Phase Locking (2PL)
▼

Fase de expansión: se toman todos los locks a usar. Fase de contracción: se liberan todos los locks (no se pueden tomar nuevos). Garantiza propiedad serializable. Pueden ocurrir deadlocks. En Strict 2PL, la contracción ocurre recién después del commit.

Control de Concurrencia
Concurrencia Optimista
▼

El proceso modifica los archivos sin ningún control, esperando que no haya conflictos. Al commitear, se verifica si el resto de las transacciones modificó los mismos archivos. Si es así, se aborta la transacción. Ventajas: Libre de deadlocks. Desventajas: Rehacer todo puede ser costoso bajo alta carga.

Ejemplo Práctico: RustiShop

🛒
RustiShop: El proceso de checkout requiere conectarse con RustiPago, RustiStock y RustiEnvios. Debe ejecutarse atómicamente: si cualquier servicio falla o rechaza, toda la operación se cancela. Solución: implementar 2PC entre los tres servicios.
16 · Distribuido IV

Ambientes Distribuidos

Modelo formal para describir sistemas distribuidos: entidades, eventos, comportamiento colectivo y métricas de costo.

Entidades

Entidad

Es la unidad de cómputo de un ambiente informático distribuido. Puede ser un proceso, un procesador, etc. Cada entidad tiene acceso de lectura/escritura a una memoria local (no compartida), capacidad de procesamiento local, comunicación via mensajes, y un reloj local.

Eventos Externos

La entidad es reactiva: solo responde a eventos externos. Los posibles eventos son:

  • Llegada de un mensaje — enviado por otra entidad
  • Activación del reloj — el reloj local dispara una alarma
  • Impulso espontáneo — el único evento que viene de fuera del sistema

Reglas y Comportamiento

Acción

Secuencia finita e indivisible de operaciones. Es atómica: se ejecuta sin interrupciones.

Regla

Relación entre evento y estado: estado × evento → acción. Para cada posible evento y estado debe existir una única regla.

Comportamiento B(x)

Conjunto de todas las reglas que obedece una entidad x. También llamado protocolo o algoritmo distribuido.

Comportamiento Homogéneo

Cuando todas las entidades del sistema tienen el mismo comportamiento: ∀x, y ∈ E, B(x) = B(y). Todo comportamiento colectivo puede transformarse en homogéneo.

Restricciones del Sistema

TipoRestricción
ConfiabilidadEntrega garantizada: cualquier mensaje enviado será recibido intacto. Parcial: no ocurrirán fallas. Total: no han ocurrido ni ocurrirán fallas.
TemporalDelays acotados (existe constante Δ), delays unitarios, relojes sincronizados.
Orientación localUna entidad puede distinguir qué vecino le envía un mensaje y puede enviar a un vecino específico.

Costo y Complejidad

📨
Costo de Mensajes (M)
Cantidad de transmisiones y carga de trabajo por entidad y de transmisión.
⏱
Tiempo de Ejecución
Tiempo total del protocolo. Tiempo ideal: medido bajo delays unitarios y relojes sincronizados.

Tipos de Conocimiento Local

  • Información métrica: datos numéricos sobre la red (n = |V|, m = |E|, diámetro, etc.)
  • Propiedades topológicas: el grafo es un anillo, es acíclico, etc.
  • Mapas topológicos: un mapa de la vecindad de la entidad hasta una distancia d (ej: matriz de adyacencia)
ℹ
Principio clave: En ausencia de fallas, el conocimiento no puede perderse. Los eventos disparan acciones que pueden generar nuevos eventos, los cuales ocurrirán siempre en un tiempo futuro Future(t).
17 · Distribuido V

Deadlocks Distribuidos

En sistemas distribuidos, los deadlocks son más difíciles de detectar porque no existe una vista global del estado. Se necesitan algoritmos especiales para detectarlos y prevenirlos.

Detección

Algoritmo
Centralizado: Grafo de uso de recursos
▼

El proceso coordinador mantiene el grafo de uso de recursos. Los procesos envían mensajes al coordinador cuando obtienen/liberan un recurso, y el coordinador actualiza el grafo. Problema: los mensajes pueden llegar desordenados y generar falsos deadlocks. Solución: usar timestamps globales (algoritmo de Lamport) para ordenar los mensajes.

Algoritmo
Distribuido: Probe Messages
▼

Cuando un proceso debe esperar por un recurso, envía un probe message al proceso que lo tiene. El mensaje contiene: id del proceso bloqueado, id del emisor, id del destinatario. Al recibir el mensaje, el proceso lo reenvía a quienes tienen el recurso que necesita. Si el mensaje llega al proceso original, tenemos un ciclo en el grafo → deadlock detectado.

Prevención mediante Timestamps

Wait-Die

Si el proceso tiene timestamp menor (más viejo), espera. Si no (es más joven), aborta la transacción y vuelve a intentar.

Wound-Wait

Si el proceso tiene timestamp menor (más viejo), aborta la transacción del proceso que tiene el recurso (lo "hiere"). Si no, espera.

ℹ
Ambos algoritmos usan timestamps globales de Lamport asignados al inicio de cada transacción. Ambos garanteízan ausencia de deadlock pero con diferentes trade-offs: Wait-Die favorece a los más viejos sin abortar a otros; Wound-Wait puede abortar transacciones activas.
18 · Extra · GPU

CUDA & Programación GPU

Las GPUs tienen miles de núcleos simples optimizados para operaciones masivamente paralelas. CUDA es el modelo de programación de NVIDIA para aprovechar este poder de cómputo.

GPU vs CPU

CPU — Low Latency Processor
  • Diseñada para minimizar la latencia
  • Caches grandes (L1, L2, L3)
  • Lógica complicada (pipeline, branch prediction)
  • Pocos núcleos potentes, context switch caro
  • Ideal para tareas secuenciales y código con mucha lógica
GPU — High Throughput Processor
  • Diseñada para maximizar el throughput
  • Más transistores en ALUs, caches pequeños
  • Los tiempos de espera de RAM se diluyen intercalando threads
  • Context switch muy económico (hasta uno por clock cycle)
  • Ideal para datos masivos con operaciones simples

SIMT — Single Instruction, Multiple Threads

Warp

La ejecución se schedulea en warps de 32 hilos. Como todos los hilos ejecutan el mismo código, comparten un único program counter (hasta arquitectura Volta). Cada hilo trabaja en un "lane" SIMD. El contexto de ejecución de cada warp se mantiene en el chip para hacer interleaving.

⚠
Branch Divergence: Cuando el código tiene saltos condicionales, el warp debe ejecutar todas las variantes aplicando una máscara de threads. La máxima eficiencia se logra cuando todos los threads del warp convergen (ejecutan el mismo camino).

Jerarquía de Threads

G
Grid

El programa se divide en un grid de bloques independientes. Representa el total del trabajo a realizar.

B
Block of Threads

Cada bloque tiene N threads. Los bloques se mantienen en el mismo SM (Streaming Multiprocessor) al ejecutarse. Óptima performance cuando cantidad de threads es múltiplo de 32.

T
Thread

Unidad mínima de ejecución. Se identifica con threadIdx.x dentro del bloque. Índice global: blockIdx.x * blockDim.x + threadIdx.x

19 · Extra · Testing

Testing Concurrente

Testear código concurrente es más difícil que el secuencial: los bugs dependen del orden de ejecución. Rust provee herramientas específicas para hacer estos tests más deterministas y confiables.

Testing en Rust

🏷
#[cfg(test)]
El módulo solo se compila cuando se ejecuta cargo test. También existe #[cfg(not(test))].
✅
#[test]
Marca funciones que son tests. Las macros assert_eq! son wrappers sobre panic().
⚡
Concurrencia por defecto
Por defecto los tests se corren de forma concurrente. Usar --test-threads=1 para secuencial.
🏗
SOLID para testabilidad
El diseño determina la testeabilidad. Dependency Inversion permite inyectar mocks fácilmente.

Crate mockall

La herramienta más completa para mocks en Rust. Permite generar automáticamente implementaciones mock de traits.

mockall
Funcionalidades principales
▼

#[automock]: Genera mocks automáticamente de traits. El mock implementa el trait con comportamiento controlable en tests.

Mocks de métodos estáticos: Permite mockear funciones estáticas y constructores, que normalmente son difíciles de reemplazar.

#[double]: Reemplaza una implementación real en tiempo de test sin cambiar el resto del código. Alternativa manual: #[cfg(not(test))] para la implementación real y #[cfg(test)] para la mock.

lazy_static para singletons de mocks: En algunos casos, se necesita acceder a la instancia del mock desde distintos lugares del test. lazy_static permite crear "singletons" controlables.

Loom — Model Checking para Concurrencia

Loom

Herramienta para testear programas concurrentes. Ejecuta los tests muchas veces, permutando las posibles ejecuciones concurrentes según el modelo de memoria C11. Usa técnicas de reducción de estados para evitar explosión combinatoria. Permite cubrir todos los posibles escenarios de ejecución.

Técnicas para Race Conditions

El Race Condition Game es un patrón de ejemplo: N threads duermen por tiempos aleatorios; el primero en despertar setea su ID en un mutex. Para testearlo:

  • Template methods: minimizar cambios de código conservando la esencia del algoritmo
  • Reemplazar sleeps no controlados por channels controlables desde el test
  • Loom: cubre todos los posibles interleavings automáticamente

Lenguajes para Concurrencia

Más allá de Rust, existen lenguajes con primitivas de concurrencia distintas. Cada uno toma decisiones de diseño radicalmente diferentes para resolver los mismos problemas.

🐹
Go
Goroutines + channels. "No comuniques compartiendo memoria; compartí memoria comunicando."
💧
Elixir
Actores sobre BEAM VM (Erlang). Tolerancia a fallos: "Let it crash".
🌀
Clojure
Dialecto de Lisp en JVM. Software Transactional Memory (STM).
📐
Julia
Scientific computing. SIMD nativo, green threads, computación distribuida.

Go — Goroutines

Goroutine

Green thread administrada por el runtime de Go. Stack inicial de solo 2KB (puede crecer dinámicamente). Cualquier función puede ejecutarse concurrentemente con la palabra reservada go. Se planifica sobre un pool de GOMAXPROCS threads del OS.

go// Lanzar una goroutine
go myFunction()

// Crear un canal
ch := make(chan int)
go func() { ch <- 42 }()
val := <-ch

// Select: escuchar varios canales
select {
case v := <-ch1: process(v)
case v := <-ch2: process(v)
}

Elixir — Actores sobre BEAM

Procesos Ligeros

Los procesos de Elixir son extremadamente baratos. El scheduler propio usa un pool de runners igual a la cantidad de CPUs.

Let it Crash

Filosofía de tolerancia a fallos: dejar que los procesos fallen y que un Supervisor los reinicie automáticamente.

OTP — GenServer

Open Telecom Platform: protocolo estandarizado para definir un proceso ("actor") extendiendo GenServer.

Funcional Puro

Immutabilidad por defecto. Pattern matching potente. Usado en producción por Discord y WhatsApp (Erlang).

Clojure — Software Transactional Memory

STM (Software Transactional Memory)

Modelo de concurrencia que trata la memoria compartida como transacciones de base de datos. Las operaciones se ejecutan especulativamente y se reintentan si hay conflictos. Libre de deadlocks. Atlassian usa Clojure para el backend de su realtime collaboration.

clojure; Referencia mutable
(def counter (ref 0))

; Transacción STM — atómica y aislada
(dosync
  (alter counter inc))

; Agent — actualización asíncrona
(def ag (agent 0))
(send ag inc)
