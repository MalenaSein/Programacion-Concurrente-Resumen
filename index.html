01 · Introducción

Programación Concurrente

Fundamentos y vocabulario esencial. La concurrencia no es igual a paralelismo — el resultado puede variar según el orden en que se intercalan las instrucciones.

Definiciones clave

Programa

Conjunto de datos, asignaciones e instrucciones de control de flujo que compilan a instrucciones de máquina. Se ejecutan secuencialmente.

Programa Concurrente

Conjunto finito de procesos secuenciales que pueden ejecutarse en paralelo. Su ejecución resulta del intercalado arbitrario de instrucciones atómicas de cada proceso.

Instrucción Atómica

Una instrucción que se ejecuta de principio a fin sin interrupciones, o directamente no se ejecuta. No hay estado intermedio observable.

ℹ
Threads vs Procesos: Los procesos tienen memoria separada. Los threads ("procesos ligeros") comparten la memoria del proceso pero cada uno tiene su propio stack, Program Counter y registros.

Los 5 Modelos de Concurrencia

1. Estado Mutable Compartido

Memoria compartida con acceso serializado mediante locks. Solo un proceso a la vez puede ejecutar ciertas regiones críticas.

2. Fork-Join

El cómputo se divide en sub-tareas independientes que se ejecutan en paralelo y luego se combinan. Ideal para divide-and-conquer.

3. Canales / Mensajes

Los procesos se comunican enviando mensajes a través de canales tipados. No hay memoria compartida directa. Más seguro.

4. Programación Asincrónica

Tareas cooperativas que se intercalan. El cambio de tarea solo ocurre en puntos específicos (await). Ideal para I/O-bound.

5. Actores

Entidades con estado privado que solo se comunican por mensajes. Cada actor procesa mensajes de a uno, sin compartir estado.

¿Cuándo usar cuál?

CPU-bound → Fork-Join / Vectorización. I/O-bound → Async. Muchos eventos concurrentes → Actores. Estado compartido simple → Locks.

Problemas clásicos de la concurrencia

🏁
Race Condition
El resultado depende del orden de ejecución de los threads. El programa se comporta diferente cada vez.
🔒
Deadlock
Dos o más procesos bloqueados mutuamente esperando recursos que el otro tiene. Nadie puede avanzar.
🍽
Starvation
Un proceso nunca obtiene el recurso que necesita porque otros siempre tienen prioridad.
🔄
Busy Wait
Un proceso espera una condición haciendo polling continuo en un loop, consumiendo CPU innecesariamente.

🎯 Quiz — Unidad 1

¿Cuál es la diferencia entre un programa y un programa concurrente?
  • Un programa concurrente es más rápido siempre
  • Un programa es secuencial; un programa concurrente es un conjunto finito de procesos secuenciales que pueden ejecutarse en paralelo con intercalado arbitrario
  • Un programa concurrente no tiene instrucciones atómicas
  • No hay diferencia, son sinónimos
Un programa secuencial ejecuta instrucciones una tras otra. Un programa concurrente está compuesto por varios procesos secuenciales cuya ejecución resulta del intercalado arbitrario de sus instrucciones atómicas.
¿Cuál es la diferencia entre concurrencia y paralelismo?
  • Son lo mismo, solo cambia el nombre según el lenguaje
  • La concurrencia es sobre la estructura (múltiples procesos que se intercalan); el paralelismo es sobre la ejecución simultánea en múltiples procesadores
  • El paralelismo es siempre más eficiente que la concurrencia
  • La concurrencia requiere múltiples CPUs obligatoriamente
Concurrencia es un concepto de diseño: múltiples procesos que progresan y se intercalan. Puede ocurrir incluso en una sola CPU (multitasking). Paralelismo implica ejecución física simultánea en múltiples procesadores.
Un deadlock ocurre cuando…
  • Un proceso consume demasiada CPU
  • Dos procesos tienen race conditions
  • Dos o más procesos están bloqueados mutuamente esperando recursos que el otro tiene, y ninguno puede avanzar
  • Un proceso entra en un loop infinito
Deadlock: proceso A espera un recurso que tiene B, y B espera un recurso que tiene A. Ninguno puede liberar lo que tiene porque está bloqueado esperando al otro.
¿Cuál modelo de concurrencia es más adecuado para I/O-bound (muchas conexiones de red)?
  • Fork-Join
  • Estado mutable compartido con RwLock
  • Programación asincrónica (async/await)
  • SIMD/Vectorización
Async es ideal para I/O-bound porque permite que un thread atienda miles de conexiones: cuando una espera una respuesta de red, el thread atiende otra. Fork-Join es para CPU-bound con tareas independientes.
¿Qué garantiza una instrucción atómica?
  • Que se ejecuta en menos de un ciclo de reloj
  • Que se ejecuta de principio a fin sin interrupciones, o directamente no se ejecuta. No hay estado intermedio observable.
  • Que varios procesos pueden ejecutarla simultáneamente
  • Que no accede a memoria compartida
Una instrucción atómica no puede ser interrumpida a la mitad. El sistema puede elegir ejecutarla completa o no ejecutarla — no hay estado intermedio observable por otros procesos.
02 · Lenguaje Base

Introducción a Rust

Rust es el lenguaje base. Sus pilares son velocidad, seguridad y concurrencia. Muchos errores que en C/C++ aparecen en runtime, en Rust se detectan en tiempo de compilación.

Variables e inmutabilidad

Las variables son inmutables por defecto. Hay que pedir mutabilidad explícitamente con mut.

let t = true;  // inmutable
// t = false; // ERROR

let mut punto = (1u8, 2u8);
punto = (4, 3);  // OK, es mutable

let numero: i32 = 42; // tipo explícito

Tipos de datos

Enteros con signo

i8, i16, i32, i64 — El número indica los bits. isize depende de la arquitectura.

Enteros sin signo

u8, u16, u32, u64 — Solo positivos. usize para índices y tamaños.

Flotantes y bool

f32, f64. bool = 1 byte. char = Unicode (4 bytes).

Compuestos

Arrays: [u8; 3]. Slices: arr[3..5]. Tuplas: (char, u8). Dinámicos: Vec<T>, String.

Funciones y Closures

fn sumar_uno(a: i32) -> i32 {
    a + 1  // sin ; = retorno implícito
}

let plus_one = |a| { a + 1 }; // closure

fn map42(mapper: fn(i32) -> i32) -> i32 {
    mapper(42) // función de orden superior
}

Structs y Traits

struct NumeroImaginario(f64, f64); // tuple struct

impl NumeroImaginario {
    fn new(r: f64, i: f64) -> Self { NumeroImaginario(r, i) }
    fn modulo(&self) -> f64 { (self.0*self.0 + self.1*self.1).sqrt() }
}

trait MagnitudVectorial { fn norma(&self) -> f64; }

impl MagnitudVectorial for NumeroImaginario {
    fn norma(&self) -> f64 { self.modulo() }
}

Enums y Pattern Matching

enum Message {
    Fire,
    Move { x: i32, y: i32 },
    Say(String),
}

match m {
    Message::Fire => println!("Fire"),
    Message::Move { x:_, y } if y > 10 => println!("Arriba"),
    Message::Move { x, y } => println!("{} {}", x, y),
    Message::Say(msg) => println!("{}", msg),
};

Option y Result — manejo de errores sin excepciones

⚠
Rust NO tiene null ni excepciones. Usa Option para valores que pueden no existir y Result para operaciones que pueden fallar. El compilador te obliga a manejar ambos casos.
// Option<T> — puede ser Some(valor) o None
fn dividir(n: f64, d: f64) -> Option<f64> {
    if d == 0.0 { None } else { Some(n / d) }
}

// Result<T,E> — puede ser Ok(valor) o Err(error)
fn abrir(path: String) -> Result<String, String> {
    Err(String::from("no existe"))
}

d.unwrap()      // PANIC si es None/Err
d.map(|n| n * 3) // transforma si existe
d?               // propaga el error (el operador ?)

🎯 Quiz — Unidad 2

¿Qué significa que las variables en Rust son inmutables por defecto?
  • No se pueden declarar en el stack
  • No se puede reasignar su valor sin usar mut explícitamente — el compilador lo prohíbe
  • Solo aplica a tipos primitivos como i32
  • Las variables se almacenan en ROM
En Rust, let x = 5 crea una variable inmutable. Para poder reasignar se necesita let mut x = 5. Esto no es una convención sino una restricción del compilador.
¿Cuál es la diferencia entre Option<T> y Result<T,E>?
  • No hay diferencia, se pueden usar indistintamente
  • Option modela un valor que puede o no existir (Some/None); Result modela una operación que puede fallar con información del error (Ok/Err)
  • Result solo se usa para errores de I/O
  • Option es más eficiente que Result en memoria
Option = "el valor puede no existir" (ej: buscar en un mapa). Result = "la operación puede fallar y quiero saber por qué" (ej: abrir un archivo). Ambos obligan al programador a manejar el caso negativo.
¿Qué hace el operador ? en Rust?
  • Es syntax sugar para unwrap() que hace panic
  • Si el valor es Err o None, retorna ese error propagándolo a la función llamadora; si es Ok/Some, extrae el valor
  • Convierte un Result en Option automáticamente
  • Solo funciona en funciones marcadas como async
El operador ? es propagación de errores. Equivale a: si es Ok(v) → devuelve v; si es Err(e) → return Err(e.into()). Permite escribir código de manejo de errores de forma concisa.
¿Cuál es el uso correcto de un trait en Rust?
  • Es igual a una clase abstracta de Java con implementación heredada
  • Define comportamiento compartido (métodos) que distintos tipos pueden implementar; similar a una interfaz pero puede tener implementaciones default
  • Solo se usa para sobrecarga de operadores
  • Reemplaza a los enums para modelar polimorfismo
Los traits definen un conjunto de métodos que un tipo debe implementar. Permiten polimorfismo sin herencia. Por ejemplo, el trait Display define cómo un tipo se convierte a string para impresión.
¿Qué es una closure en Rust y en qué se diferencia de una función?
  • Es idéntica a una función, solo que se declara con sintaxis diferente
  • Es una función anónima que puede capturar variables del entorno donde fue creada
  • Es una función que solo puede recibir un parámetro
  • No puede pasarse como parámetro a otras funciones
Las closures capturan su entorno: let factor = 3; let mult = |x| x * factor; — mult captura factor. Las funciones regulares no pueden hacer esto. Las closures son fundamentales para iteradores y callbacks.
03 · Rust Avanzado

Ownership & Borrowing

El sistema de ownership es lo que hace a Rust único. Garantiza memory safety sin garbage collector, eliminando clases enteras de errores en tiempo de compilación.

Las 3 Reglas de Ownership

1️⃣
Un único dueño
Cada valor tiene exactamente un owner. No puede haber dos owners del mismo valor simultáneamente.
2️⃣
Move = cambiar dueño
Al asignar a otra variable o pasar a una función, la variable original queda inválida. Ya no se puede usar.
3️⃣
Drop automático
Cuando el owner sale del scope, se llama automáticamente al trait Drop. Sin free() manual.

Borrowing: prestar referencias

let mut ariel = Persona { edad: 37 };

// ✓ Múltiples referencias inmutables OK
let r1 = &ariel;
let r2 = &ariel;
println!("{:?} {:?}", r1, r2);

// ✓ Una referencia mutable
let r3 = &mut ariel;
r3.edad += 1;

// ✗ NO se puede mutable + inmutable al mismo tiempo
// let r4 = &ariel; // ERROR si r3 sigue activo
✓
Por qué esto previene data races: Solo puede existir UNA referencia mutable a la vez. Mientras existe, no puede haber referencias inmutables. El compilador verifica esto en compile-time → imposible tener data races.

Punteros Inteligentes

Box<T>

Puntero a heap con un único owner. Para datos de tamaño desconocido en compile-time o recursión.

Rc<T>

Reference Counting. Múltiples owners inmutables. Solo un thread.

Arc<T>

Atomic RC. Como Rc pero thread-safe. Para compartir entre múltiples threads.

Arc<RwLock<T>>

Patrón típico: compartir entre threads con acceso mutable controlado por un lock.

Traits Send y Sync

Send

El ownership puede transferirse entre threads. Casi todos los tipos son Send. Excepción: Rc<T>.

Sync

Seguro ser referenciado desde múltiples threads. T es Sync si &T es Send.

Threads en Rust

use std::thread;

let handle = thread::spawn(move || {
    // 'move' = el closure toma ownership de las vars capturadas
    println!("soy el thread hijo");
});

handle.join().unwrap(); // espera a que el hijo termine
ℹ
¿Por qué move? El thread hijo puede vivir más que el scope del padre. move fuerza al closure a ser dueño de todo lo que usa, evitando referencias inválidas.

🎯 Quiz — Unidad 3

¿Qué sucede cuando pasás una variable a una función sin usar referencias en Rust?
  • Se hace una copia automática del valor
  • El ownership se transfiere (move) a la función; la variable original queda inválida y no puede usarse más
  • Se crea una referencia inmutable automáticamente
  • Rust lanza un error de compilación siempre
En Rust los tipos no-Copy (como String, Vec) hacen move al pasarse a funciones. El ownership pasa a la función y la variable original no puede usarse. Para evitarlo, se usa borrowing (&) o .clone().
¿Cuántas referencias mutables se pueden tener a un valor al mismo tiempo?
  • Ilimitadas, siempre que no haya lecturas simultáneas
  • Exactamente una, y mientras existe esa referencia mutable no puede haber ninguna referencia inmutable
  • Tantas como núcleos tenga el procesador
  • Dos: una de lectura y una de escritura simultáneamente
Regla fundamental del borrow checker: O muchas referencias inmutables (&T), O exactamente una mutable (&mut T), nunca ambas al mismo tiempo. Esto elimina data races en compile-time.
¿Para qué sirve Arc<Mutex<T>> en Rust?
  • Para tener múltiples referencias inmutables en un solo thread
  • Para compartir estado mutable entre múltiples threads de forma segura: Arc maneja la vida útil compartida, Mutex garantiza acceso exclusivo
  • Arc reemplaza a Mutex cuando hay muchos lectores
  • Es equivalente a usar unsafe con un puntero raw
Arc (Atomic Reference Counting) permite múltiples owners en distintos threads. Mutex garantiza que solo un thread accede al valor a la vez. Juntos son el patrón estándar para estado compartido mutable entre threads.
¿Qué diferencia hay entre Rc<T> y Arc<T>?
  • Arc usa más memoria pero es igual en todo lo demás
  • Rc no implementa Send ni Sync (solo para un thread); Arc usa operaciones atómicas para el conteo y es thread-safe
  • Rc es para heap, Arc es para stack
  • No hay diferencia funcional, Arc es solo un alias moderno
Rc usa conteo de referencias no atómico → más rápido pero solo un thread. Arc usa operaciones atómicas → algo más lento pero seguro entre threads. El compilador prohíbe usar Rc entre threads (no implementa Send).
¿Por qué se usa move en el closure de thread::spawn?
  • Para que el thread sea más rápido al arrancar
  • Es obligatorio sintácticamente, no tiene significado semántico
  • Para que el closure tome ownership de las variables capturadas, evitando referencias a datos que podrían ser eliminados antes de que el thread termine
  • Para poder modificar las variables capturadas dentro del thread
Sin move, el closure captura por referencia. Pero el thread hijo puede vivir más que el scope del padre, dejando referencias dangling. Con move, el thread es dueño de los datos y los puede usar por todo su lifetime.
04 · Paralelismo

Fork-Join

Modelo donde el cómputo se divide recursivamente en sub-tareas independientes que se ejecutan en paralelo y luego se combinan. Paper original de Melvin Conway, 1963.

Propiedades del modelo

🔀
División Recursiva
Sub-cómputos independientes entre sí → pueden ejecutarse en paralelo sin coordinación.
🎲
Determinístico
Threads aislados. El resultado es el mismo independientemente de las diferencias de velocidad.
⚡
Performance ideal
t_secuencial / N_threads. Puede variar por tamaño de tareas y costo de combinar.
⚠️
Desventaja
Requiere que las unidades de trabajo sean aisladas. No funciona si comparten estado mutable.

Work Stealing — el scheduling inteligente

Cuando una tarea termina, su thread puede robar trabajo de otros threads que están ocupados. Cada thread tiene una cola de doble extremo (deque):

  • Encola subtareas al final de su propia deque
  • Toma trabajo del final de la propia (LIFO — localidad de caché)
  • Si está vacía, roba del inicio de una cola aleatoria (FIFO — minimiza contención)

Rayon — Fork-Join de alto nivel

use rayon::prelude::*;

// Cambiar iter() por par_iter() = magia paralela
giant_vector.par_iter().for_each(|value| {
    do_thing_with_value(value);
});

// Dos tareas en paralelo
let (v1, v2) = rayon::join(fn1, fn2);

// Map + Reduce paralelo
let s = ['a', 'b', 'c']
    .par_iter()
    .map(|c| format!("{}", c))
    .reduce(|| String::new(), |mut a, b| { a.push_str(&b); a });

Crossbeam — scoped threads

use crossbeam;
let data = vec![1, 2, 3];

crossbeam::scope(|s| {
    // Podemos tomar referencias al stack porque scope garantiza
    // que todos los threads terminan antes de que scope retorne
    s.spawn(|_| { println!("{:?}", data); });
}).unwrap();

🎯 Quiz — Unidad 4

¿Cuál es la propiedad más importante que deben tener las subtareas en Fork-Join?
  • Deben tener el mismo tamaño exacto
  • Deben ser independientes entre sí — no compartir estado mutable — para poder ejecutarse en paralelo sin coordinación
  • Deben ejecutarse en el mismo thread para maximizar cache
  • Deben terminar todas antes de que empiece cualquier otra
La independencia es el requisito fundamental de Fork-Join. Si las subtareas comparten estado, necesitarían sincronización y perderían las garantías de determinismo y ausencia de race conditions del modelo.
¿Qué hace Work Stealing cuando el thread propio se queda sin trabajo?
  • Espera bloqueado hasta que el coordinador le asigne una nueva tarea
  • Intenta robar una tarea del inicio (FIFO) de la cola de un thread aleatorio que esté ocupado
  • Crea nuevas subtareas dividiendo las tareas pendientes de otros
  • Termina su ejecución y reduce el pool de threads
Work stealing: cada thread tiene una deque. Trabaja en sus propias tareas desde el final (LIFO, mejor cache). Si se queda vacía, roba del inicio de la deque de otro thread (FIFO, minimiza contención en el extremo que el dueño usa).
¿Qué hace rayon::join(fn1, fn2)?
  • Ejecuta fn1 y luego fn2 secuencialmente
  • Ejecuta fn1 y fn2 potencialmente en paralelo y retorna una tupla con ambos resultados cuando ambas terminan
  • Combina (join) el resultado de fn1 y fn2 con una operación de merge por defecto
  • Crea dos threads del SO nuevos para ejecutar cada función
rayon::join divide el trabajo en dos ramas que Rayon puede ejecutar en paralelo usando su pool de workers. Retorna la tupla (resultado1, resultado2) cuando ambas terminan. Internamente usa work stealing.
¿Por qué Fork-Join es determinístico?
  • Porque Rayon garantiza un orden de ejecución específico
  • Porque los threads están aislados y no comparten estado — el resultado es el mismo independientemente del orden de ejecución de los threads
  • Porque usa locks para serializar el acceso a los resultados
  • Porque siempre crea el mismo número fijo de threads
La independencia de las subtareas garantiza determinismo: no importa qué thread ejecutó qué parte, el resultado combinado siempre es el mismo. No hay race conditions porque no hay estado compartido.
¿Cuándo NO es apropiado usar Fork-Join?
  • Para ordenar un array de 10 millones de elementos
  • Para aplicar un filtro de imagen a 1000 fotos independientes
  • Para implementar un servidor web que atiende miles de conexiones simultáneas de red
  • Para calcular la suma de un vector de 1 millón de enteros
Un servidor web es I/O-bound: los threads pasarían casi todo el tiempo esperando respuestas de red. Fork-Join es CPU-bound. Para un servidor web es mucho más eficiente usar async/await que escala a miles de conexiones con pocos threads.
05 · Datos Masivos

MapReduce, Dremel & SIMD

Paradigmas para procesamiento masivo de datos y operaciones vectoriales que aprovechan el hardware moderno.

MapReduce

Paper de Google 2004. El usuario especifica dos funciones:

Map

Toma pares clave/valor y genera pares clave/valor intermedios. Se ejecuta en paralelo sobre distintas particiones.

Reduce

Combina todos los valores intermedios asociados con la misma clave. Produce el resultado final.

Word Count (el "Hello World" de MapReduce): Map extrae frecuencias locales de palabras. Reduce combina todas las frecuencias en una tabla global.

Dremel / BigQuery

Sistema de consultas en tiempo casi real. Una consulta se propaga hacia abajo en un árbol de nodos y se ensambla combinando respuestas de los niveles inferiores. Permite queries sobre petabytes en segundos.

Vectorización y SIMD

SIMD — Single Instruction, Multiple Data

Instrucciones de CPU que operan sobre vectores de datos en paralelo. Cada registro es un vector de tamaño fijo (128-512 bits) dividido en N "lanes". Instrucciones: MMX, SSE, AVX en x86; NEON en ARM.

Operaciones Verticales

Entre distintos registros en el mismo lane. Las más rápidas. Suma x₀+y₀, x₁+y₁, ... simultáneamente.

Operaciones Horizontales

Reducen un vector a un escalar. Más lentas porque combinan datos de distintos lanes.

Un registro de 512 bits puede alojar: 64 enteros de 8 bits, 16 enteros de 32 bits, u 8 flotantes de 64 bits — todos procesados con una sola instrucción.

GPU vs CPU

CPU — Low Latency

Minimizar latencia. Caches grandes, lógica compleja (pipeline, branch prediction). Pocos núcleos potentes. Context switch caro (~microsegundos).

GPU — High Throughput

Maximizar throughput. Más transistores en ALUs, miles de núcleos simples. Context switch muy barato (hasta uno por clock cycle).

🎯 Quiz — Unidad 5

En MapReduce, ¿cuál es la responsabilidad de la función Map?
  • Combinar todos los valores de una clave en un resultado final
  • Procesar pares clave/valor de entrada para generar pares clave/valor intermedios, ejecutándose en paralelo sobre distintas particiones de datos
  • Distribuir los datos entre los nodos del cluster
  • Ordenar los resultados finales del cómputo
Map opera sobre fragmentos independientes de datos en paralelo, transformando cada entrada en pares intermedios. En Word Count, Map toma líneas de texto y emite (palabra, 1) por cada palabra encontrada.
¿Qué significa SIMD?
  • System Integrated Memory Design
  • Single Instruction, Multiple Data — una instrucción que opera sobre múltiples datos en paralelo usando registros vectoriales de la CPU
  • Synchronized Instruction Memory Dispatch
  • Sequential Instruction Multiple Division
SIMD permite que una sola instrucción opere sobre N datos simultáneamente. Un registro de 256 bits puede sumar 8 floats de 32 bits en una sola operación. Instrucciones: SSE, AVX en x86; NEON en ARM.
¿Cuál es la diferencia entre operaciones SIMD verticales y horizontales?
  • Verticales operan entre distintos registros en el mismo lane (más rápidas); horizontales reducen un vector a un escalar combinando lanes (más lentas)
  • Vertical = en GPU; horizontal = en CPU
  • No hay diferencia de performance, es solo nomenclatura
  • Horizontales son las operaciones entre registros y verticales son internas al registro
Vertical: suma x[0]+y[0], x[1]+y[1], ... en paralelo — muy eficiente. Horizontal: suma x[0]+x[1]+x[2]+... para obtener un escalar — requiere combinar datos de lanes distintos, más lento.
¿Por qué las GPUs tienen context switch muy barato comparado con CPUs?
  • Porque tienen menos threads en total
  • Porque están diseñadas para maximizar throughput: mantienen el estado de miles de threads simultáneamente y pueden cambiar entre ellos en uno o pocos ciclos de clock
  • Porque no tienen sistema operativo que gestione los cambios
  • Porque sus threads no tienen stack propio
GPU: miles de threads simples con estado mínimo. Cambio de contexto puede ser un clock cycle — la GPU los intercala constantemente para ocultar latencias de RAM. CPU: pocos threads complejos con estado grande (registros, caché, predictor). Context switch cuesta microsegundos.
Un registro SIMD de 512 bits ¿cuántos enteros de 32 bits puede procesar simultáneamente?
  • 8
  • 32
  • 16
  • 64
512 bits / 32 bits por entero = 16 enteros. Con AVX-512 se puede sumar, multiplicar o comparar 16 enteros de 32 bits con una sola instrucción. También puede alojar 8 doubles (64 bits) o 64 bytes (8 bits).
06 · Asincronismo

Async / Await & Futures

Modelo de concurrencia cooperativo. Ideal para operaciones I/O-bound donde los threads pasarían la mayor parte del tiempo esperando.

❌ NO usar async

Factorial, producto de matrices, filósofos, estado compartido. Son CPU-bound. Usá Fork-Join.

✓ SÍ usar async

Consultar APIs, leer archivos, servir requests HTTP. Son I/O-bound. El thread puede atender otras tareas mientras espera.

Futures — el concepto central

Future

Un valor que representa un cómputo que puede no estar completo todavía. El método poll nunca bloquea: retorna Ready(valor) si terminó, o Pending si no.

La metáfora de la piñata: Lo único que se puede hacer con un Future es "golpearlo con poll" hasta que caiga el valor. Cada vez que se llama poll, el future avanza todo lo que puede.

trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),   // terminó, acá está el valor
    Pending,    // todavía no terminó
}

Funciones async y await

// async fn → retorna un Future inmediatamente
// El cuerpo NO se ejecuta hasta que se hace poll
async fn cheapo_request(host: &str, port: u16) -> Result<String> {
    let mut socket = TcpStream::connect((host, port)).await?;
    // .await suspende ESTA tarea pero libera el thread para otras
    socket.read_to_string(&mut response).await?;
    Ok(response)
}
⚠
Diferencia clave con threads: El cambio de tarea ocurre SOLO en las expresiones await. Un cómputo CPU-intensivo en async bloquearía el thread. Para eso usar spawn_blocking.

block_on, spawn y yield

// block_on: puente sync → async. Bloquea hasta Ready.
let response = task::block_on(cheapo_request("example.com", 80))?;

// spawn: crear tarea en el pool multi-thread
task::spawn(async { do_something().await });

// join!: ejecutar múltiples futures en paralelo
let (h, w) = futures::join!(hello(), world());

Pin<T>

Los futures generados por async guardan referencias a sí mismos. Si fueran movidos en memoria, esas referencias internas quedarían inválidas. Pin "fija" el future en una dirección de memoria, haciendo imposible moverlo si implementa !Unpin.

🎯 Quiz — Unidad 6

¿Qué retorna una función async fn cuando la invocás?
  • El valor final inmediatamente, bloqueando el thread hasta tenerlo
  • Un thread nuevo que ejecuta la función
  • Un Future que representa el cómputo a realizar — el cuerpo aún no ejecutó nada
  • Un Option con el valor si ya está listo, None si no
Invocar una async fn retorna inmediatamente un Future sin ejecutar el cuerpo. El cuerpo solo empieza cuando alguien hace poll sobre el Future (via .await, block_on, o spawn).
¿Qué hace .await sobre un Future?
  • Bloquea el thread completo hasta que el Future termine
  • Suspende la tarea actual liberando el thread para otras tareas, y la reanuda cuando el Future esté Ready
  • Crea un nuevo thread para ejecutar el Future
  • Convierte el Future en un valor Result
await suspende ESTA tarea (no el thread). El thread queda libre para ejecutar otras tareas async. Cuando el Future completa (ej: llega respuesta de red), el runtime reactiva la tarea desde donde estaba suspendida.
¿Por qué async es MÁS eficiente que threads para I/O-bound?
  • Porque async usa instrucciones SIMD internamente
  • Las tareas async son ~50x más rápidas de crear que threads, usan mucha menos memoria, y el context switch no requiere syscall al kernel
  • Porque async puede usar múltiples CPUs simultáneamente
  • No es más eficiente, solo tiene sintaxis más legible
Thread Linux: ~20KB stack, ~15μs creación, context switch con syscall. Tarea async: mucho menor tamaño, ~300ns creación, context switch en userspace. Para 10,000 conexiones simultáneas, threads son inviables; tasks async, trivial.
¿Cuándo NO se debe usar async en Rust?
  • Para consultar bases de datos externas
  • Para servir requests HTTP
  • Para cómputos CPU-intensivos como calcular un producto de matrices o un factorial — bloquearían el thread impidiendo a otras tasks ejecutarse
  • Para leer archivos del disco
Async es cooperativo: el cambio de tarea solo ocurre en los .await. Un loop CPU-intensivo sin .await monopoliza el thread, bloqueando todas las demás tasks. Para CPU-bound usar spawn_blocking o Fork-Join.
¿Para qué sirve Pin<T> en el contexto de Futures?
  • Para mejorar el performance fijando el Future en la caché de la CPU
  • Para evitar que un Future sea movido en memoria, ya que los async/await autogenerados contienen referencias a sí mismos que quedarían inválidas si se movieran
  • Para garantizar que el Future se ejecuta en un thread específico
  • Es solo un wrapper de seguridad sin efecto real en compiladores modernos
Un Future generado por async puede almacenar referencias a su propio estado interno. Si el Future fuera movido en memoria, esas referencias quedarían dangling. Pin garantiza que el Future permanece en la misma dirección de memoria.
07 · Sincronización I

Corrección & Locks

En programas concurrentes la salida puede depender del escenario de ejecución. Necesitamos propiedades formales para razonar sobre corrección.

Propiedades de Corrección

Safety — siempre verdadera

Exclusión mutua: dos procesos no intercalan instrucciones de la sección crítica.
Ausencia de deadlock: si dos procesos quieren entrar, alguno lo logrará.

Liveness — eventualmente verdadera

Ausencia de starvation: todo proceso que espera eventualmente obtiene el recurso.
Fairness: una instrucción habilitada eventualmente aparece en la ejecución.

La Sección Crítica

Código que accede a recursos compartidos. Reglas:

  • La sección crítica debe progresar (finalizar eventualmente)
  • La sección no-crítica no requiere progreso (puede entrar en loop infinito)
  • Solo un proceso a la vez puede ejecutar la sección crítica

Locks en UNIX

Shared Lock (Read Lock)

Más de un proceso puede tener el lock simultáneamente. Para proteger lecturas.

Exclusive Lock (Write Lock)

Solo un proceso puede tener el lock. Excluye tanto a lectores como escritores.

RwLock en Rust

use std::sync::{Arc, RwLock};

let lock = RwLock::new(1);

// Lock de lectura (múltiples lectores OK)
let n = lock.read().unwrap();
println!("{}", *n);
// Se libera automáticamente al salir del scope (RAII)

// Lock de escritura (acceso exclusivo)
let mut w = lock.write().unwrap();
*w += 1;

// Compartir entre threads: Arc + RwLock
let shared = Arc::new(RwLock::new(0i32));
let clone = shared.clone();
std::thread::spawn(move || { *clone.write().unwrap() += 1; });
⚠
Lock Envenenado (Poisoned): Si un thread hace panic mientras tiene el lock de escritura, el lock queda "poisoned". Las llamadas posteriores a read() y write() retornan Err. Esto es intencional: el estado puede haber quedado inconsistente.

🎯 Quiz — Unidad 7

¿Cuál es la diferencia entre propiedades Safety y Liveness?
  • Safety es para threads, Liveness es para procesos
  • Safety debe ser verdadera siempre (ej: exclusión mutua); Liveness debe volverse verdadera eventualmente (ej: ausencia de starvation)
  • Liveness implica safety pero no al revés
  • Son sinónimos, solo cambia en qué literatura se usa cada término
Safety = "algo malo nunca pasa" (invariante permanente). Liveness = "algo bueno eventualmente pasa". Deadlock viola safety. Starvation viola liveness. Un sistema puede tener safety sin liveness: excluye bien pero deja a alguien esperando para siempre.
¿Qué significa que un lock de UNIX sea "advisory"?
  • Los procesos pueden ignorar los locks — el kernel no los impone automáticamente; solo tienen efecto si todos los procesos cooperan
  • El lock solo dura mientras el proceso está activo
  • El kernel aconseja al proceso cuándo tomar el lock pero no lo bloquea
  • El lock es automáticamente liberado después de un timeout
Advisory locks en Unix son cooperativos: un proceso puede acceder al archivo sin respetar el lock si quiere. Solo protegen si todos los procesos involucrados verifican y respetan los locks antes de acceder.
¿Qué ocurre con un RwLock en Rust cuando un thread hace panic mientras tiene el write lock?
  • El lock se libera automáticamente y los otros threads continúan
  • El lock queda "poisoned": llamadas posteriores a read() y write() retornan Err, indicando que el estado puede ser inconsistente
  • El programa entero hace panic automáticamente
  • El lock se resetea a su estado inicial
Lock poisoning es intencional: si un thread panicked mientras modificaba datos protegidos, esos datos pueden estar en estado inconsistente. Rust señaliza esto con Err para que los demás threads decidan si pueden continuar o no.
¿Para qué sirve un shared (read) lock vs un exclusive (write) lock?
  • Son equivalentes, solo cambia el nombre por convención
  • Read lock: múltiples procesos pueden leerlo simultáneamente. Write lock: solo un proceso a la vez, excluyendo tanto lectores como escritores
  • Read lock es más rápido de obtener siempre, independientemente de la contención
  • Write lock no bloquea a los lectores, solo a otros escritores
Multiple readers, single writer: muchos pueden leer el mismo dato simultáneamente sin problema. Pero si alguien modifica (escribe), nadie más puede leer ni escribir hasta que termine, evitando lecturas de datos inconsistentes.
¿Cuáles son las 3 propiedades de corrección que debe cumplir una solución a la sección crítica?
  • Velocidad, correctitud y portabilidad
  • Exclusión mutua (no intercalar secciones críticas), ausencia de deadlock (alguno eventualmente entra), ausencia de starvation (todo proceso que espera eventualmente entra)
  • Atomicidad, consistencia y aislamiento
  • Fairness, throughput y latencia
Las tres propiedades son jerárquicas: exclusión mutua es la más fundamental (safety), luego deadlock-freedom (safety), luego starvation-freedom (liveness). Una solución puede tener las dos primeras sin la tercera.
08 · Sincronización II

Semáforos

Mecanismos de sincronización de más alto nivel que los locks. Introducidos por Dijkstra en los años 60. Un contador con dos operaciones atómicas.

Semáforo S

Tipo de dato compuesto por: V (entero no negativo = contador de recursos) y L (conjunto de procesos bloqueados). Se inicializa con k ≥ 0 y ∅.

Operaciones

wait(S) — p(S) — "tomar el recurso"
if S.V > 0:
    S.V := S.V - 1   // hay recurso, tomarlo
else:
    bloquear el proceso

Decrementa el contador. Si llega a 0, el proceso se bloquea.

signal(S) — v(S) — "liberar el recurso"
if S.L.isEmpty():
    S.V := S.V + 1   // nadie espera, liberar
else:
    despertar uno de L

Incrementa el contador. Si hay procesos esperando, despierta uno.

Invariantes del Semáforo

  • S.V ≥ 0 — El valor nunca es negativo
  • S.V = k + #signal(S) − #wait(S) — siendo k el valor inicial
  • wait() y signal() son instrucciones atómicas
  • El signal() despierta a uno de los procesos pero no está definido cuál

Problema Productor-Consumidor

Buffer Infinito (solo semáforo notEmpty)

Productor
loop forever:
  p1: append(d, buffer)
  p2: signal(notEmpty)
→
Consumidor
loop forever:
  q1: wait(notEmpty)
  q2: d ← take(buffer)

Buffer Acotado (N) — dos semáforos

SemáforoValor inicialPropósito
notEmpty(0, ∅)Bloquea al consumidor si el buffer está vacío
notFull(N, ∅)Bloquea al productor si el buffer está lleno

Barreras en Rust

use std::sync::{Arc, Barrier};
let barrier = Arc::new(Barrier::new(3)); // para 3 threads

// Cada thread llama wait() al llegar al punto de sincronización
// Todos se bloquean hasta que los 3 llegaron
let bw = barrier.wait();
if bw.is_leader() {
    println!("Todos llegaron, soy el líder");
}

🎯 Quiz — Unidad 8

Un semáforo se inicializa con valor 3. Se hacen 2 wait(). ¿Cuál es el valor actual?
  • 0
  • 1
  • 5
  • -1 (proceso bloqueado)
S.V = k + #signal - #wait = 3 + 0 - 2 = 1. El valor nunca baja de 0 — si baja a 0 y se hace otro wait, el proceso se bloquea en lugar de decrementar a negativo.
¿Qué pasa si se llama signal() en un semáforo cuando hay procesos esperando en su cola?
  • El contador V se incrementa y todos los procesos esperando se despiertan
  • Se despierta uno de los procesos de la cola S.L (no se define cuál), y el contador V NO se incrementa
  • El proceso que llama signal() se bloquea hasta que el proceso despertado termina
  • El contador V se incrementa aunque haya procesos esperando
Si S.L no está vacío: se despierta un proceso de S.L (el cuál no está definido por la especificación) y V permanece en 0. Si S.L está vacío: V se incrementa. El signal "transfiere" el recurso directamente al proceso que esperaba.
En el Productor-Consumidor con buffer acotado (N), ¿con qué valor se inicializa el semáforo notFull?
  • 0
  • 1
  • N (la capacidad máxima del buffer)
  • N/2
notFull representa los espacios libres disponibles. Inicialmente el buffer está vacío → N espacios libres → notFull = N. El productor hace wait(notFull) antes de producir (consume un espacio), y el consumidor hace signal(notFull) al consumir (libera un espacio).
¿Cuál es el invariante más importante de un semáforo?
  • S.V nunca supera su valor inicial
  • S.V ≥ 0 siempre, y S.V = k + #signal(S) − #wait(S)
  • La cantidad de procesos en S.L nunca supera S.V
  • wait() y signal() deben llamarse el mismo número de veces
S.V ≥ 0 es invariante fundamental (no puede ser negativo). La ecuación S.V = k + #signal - #wait permite razonar sobre el estado del semáforo en cualquier punto de la ejecución.
¿Para qué sirve una Barrera (Barrier) en Rust?
  • Para serializar el acceso a un recurso compartido entre threads
  • Para sincronizar N threads en un punto determinado: todos se bloquean hasta que todos llegaron, luego continúan juntos
  • Para limitar la cantidad de threads que pueden ejecutarse simultáneamente
  • Es equivalente a un semáforo binario
Barrier::new(N) crea una barrera para N threads. Cuando cada thread llama wait(), se bloquea hasta que los N threads llamaron wait(). Útil para sincronizar fases de un algoritmo paralelo (ej: todos terminaron la fase 1, ahora empiezan la fase 2).
09 · Sincronización III

Monitores & Condition Variables

Herramienta de sincronización de más alto nivel. Encapsula el estado compartido con las operaciones que lo modifican, y garantiza exclusión mutua implícitamente.

Monitor

Garantiza exclusión mutua automática para sus procedimientos. Solo un proceso ejecuta el monitor a la vez. Tiene condition variables para esperar y señalizar condiciones.

Condition Variables vs Semáforos

SemáforoMonitor (Condvar)
wait puede o NO bloquearwaitC siempre bloquea
signal siempre tiene efectosignalC no tiene efecto si la cola está vacía
signal desbloquea proceso arbitrariosignalC desbloquea el proceso del tope de la cola (FIFO)
Proceso desbloqueado continúa inmediatamenteDebe esperar a que el señalizador deje el monitor

Mutex + Condvar en Rust

use std::sync::{Arc, Condvar, Mutex};

let pair = Arc::new((Mutex::new(false), Condvar::new()));
let pair2 = pair.clone();

// Thread que ESPERA
std::thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut ready = lock.lock().unwrap();
    // ¡SIEMPRE while, nunca if! (por spurious wakeups)
    while !*ready {
        ready = cvar.wait(ready).unwrap();
    }
    println!("Condición cumplida!");
});

// Thread que SEÑALIZA
let (lock, cvar) = &*pair;
*lock.lock().unwrap() = true;
cvar.notify_all();  // o notify_one()
⚠
Spurious Wakeups: Un thread puede despertar de un wait sin que notify haya sido llamado (comportamiento permitido por POSIX). Por eso la condición SIEMPRE debe verificarse en un while, nunca en un if.

Monitores en Java

// Cada objeto Java tiene un monitor asociado
public synchronized void incrementar(int cantidad) {
    this.valor += cantidad; // exclusión mutua automática
}

// wait() libera el monitor y suspende el hilo
// notify() despierta alguno. notifyAll() despierta a todos.
// ¡También usar while, no if!

🎯 Quiz — Unidad 9

¿Cuál es la diferencia clave entre waitC (monitor) y wait (semáforo)?
  • waitC es más eficiente porque no usa syscalls
  • waitC SIEMPRE bloquea al proceso; wait de semáforo puede o no bloquear según el valor del contador
  • waitC libera el monitor automáticamente; wait no libera nada
  • Son equivalentes, solo cambia el nombre
Semáforo: si V>0, wait decrementa y continúa (NO bloquea). Monitor/Condvar: wait SIEMPRE bloquea el proceso y libera el monitor para que otros entren. Esto hace que la Condvar sea una herramienta de espera pura.
¿Por qué se debe usar while en lugar de if al verificar la condición en Condvar::wait?
  • Porque if no compila con guards de Mutex
  • Por los spurious wakeups: POSIX permite que un thread despierte de wait() sin que notify haya sido llamado, por lo que hay que re-verificar la condición
  • Para que múltiples threads puedan esperar la misma condición
  • Porque notify_all puede despertar threads que no necesitan la condición
Spurious wakeups son reales y están permitidos por POSIX. Si usás if, el thread continuaría aunque la condición sea falsa. Con while, re-verifica y vuelve a esperar si es necesario. Regla: SIEMPRE while con Condvar.
¿Qué hace signalC de un monitor si la cola de la condition variable está vacía?
  • Guarda el "crédito" para el próximo proceso que haga waitC
  • No tiene ningún efecto — el signal se pierde
  • Incrementa un contador interno de la condición
  • Lanza una excepción o error
Diferencia crucial: semáforo signal() siempre incrementa V (guarda el "crédito"). Monitor signalC() no tiene efecto si nadie espera — no guarda nada. Por eso en monitores la condición debe re-verificarse con while.
En Java, ¿qué garantiza la palabra clave synchronized en un método?
  • Que el método se ejecuta en un thread dedicado
  • Exclusión mutua: solo un thread a la vez puede ejecutar ese método en el mismo objeto; los demás se bloquean hasta que el primero termina
  • Que las variables locales del método son thread-safe
  • Que el método no puede hacer wait() ni notify()
synchronized en Java adquiere el monitor del objeto (this). Dos threads no pueden ejecutar métodos synchronized del mismo objeto simultáneamente. Para métodos estáticos, el monitor es el Class object.
¿Cuál es la diferencia entre notify() y notifyAll() en Java?
  • notify() despierta a un thread arbitrario de la cola de espera; notifyAll() despierta a todos los threads que esperan en ese objeto
  • notifyAll() es más eficiente porque evita spurious wakeups
  • notify() solo funciona con synchronized, notifyAll() funciona sin él
  • Son equivalentes en práctica ya que igual hay que usar while
notify() despierta uno (no definido cuál). notifyAll() despierta todos — más seguro pero potencialmente más caro si muchos threads esperan. En general se prefiere notifyAll() para evitar que el thread equivocado sea notificado y nadie más despierte.
10 · Formalismo

Redes de Petri

Formalismo matemático para modelar sistemas concurrentes. Permite razonar sobre propiedades como deadlock, alcanzabilidad y corrección de manera visual y rigurosa.

●
Lugar (p): estado o condición
Transición (t): evento o acción
Token (•): lugar marcado / activo

Red Ordinaria

PN = (T, P, A)

T = transiciones, P = lugares, A ⊆ (T×P) ∪ (P×T) = arcos. Una transición t está habilitada si M(p) ≥ 1 para todo p ∈ I(t). Al disparar: consume tokens de entrada, produce en salida.

Red General (con pesos)

PN = (T, P, A, W, M₀)

W: A → ℕ es la función de peso. T habilitada si M(p) ≥ W(p,t) para todo p ∈ I(t). Al disparar: M(p) ← M(p) − W(p,t) para entradas; M(p') ← M(p') + W(p',t) para salidas.

Arco Inhibidor

Se representa con flecha de punta abierta. Una transición con arco inhibidor desde p solo está habilitada si p NO tiene tokens. Usado en el problema Lector-Escritor.

Modelos Clásicos

Mutex

P2 inicializado con 1 token actúa como lock. Solo uno de los procesos puede tener ese token y entrar a la sección crítica.

Productor-Consumidor (acotado)

Lugar p₆ con N tokens representa espacios libres. La transición que produce necesita un token de p₆. La que consume lo devuelve.

Lector-Escritor

T0 (lector) tiene arco inhibidor desde "escritores activos". T2 (escritor) tiene arco inhibidor desde "lectores activos".

Cliente-Servidor

El servidor espera requests (token en "esperando"). Al recibir, procesa y devuelve respuesta, volviendo al estado de espera.

🎯 Quiz — Unidad 10

En una Red de Petri, ¿cuándo está habilitada una transición para disparar?
  • Cuando alguno de sus lugares de entrada tiene al menos 1 token
  • Cuando TODOS sus lugares de entrada tienen al menos W(p,t) tokens (donde W es el peso del arco)
  • Cuando el marcado total de la red supera un umbral
  • Cuando sus lugares de salida están vacíos
Una transición t está habilitada si y solo si M(p) ≥ W(p,t) para TODOS los p ∈ I(t). Todos sus lugares de entrada deben tener suficientes tokens según el peso del arco. Falla un lugar → transición no habilitada.
¿Qué representa un token en un lugar de una Red de Petri?
  • Un thread en ejecución
  • Que esa condición o estado está activo/verdadero en el sistema modelado
  • Un mensaje en tránsito entre procesos
  • Un lock tomado por un proceso
Los tokens representan el estado del sistema. Un lugar con token = esa condición está activa. En un modelo de mutex, el token en el lugar "libre" indica que el recurso está disponible. Al tomarlo (transición dispara), el token se mueve a "ocupado".
¿Para qué se usa un arco inhibidor en una Red de Petri?
  • Para aumentar el peso de una transición multiplicando los tokens
  • Para habilitar una transición solo cuando un lugar específico NO tiene tokens — es la negación de la condición
  • Para conectar dos transiciones directamente sin pasar por un lugar
  • Para limitar el número máximo de tokens en un lugar
Arco inhibidor (punta abierta): la transición está habilitada si el lugar tiene 0 tokens. Se usa en Lector-Escritor: la transición "escribir" tiene arco inhibidor desde "lectores activos" → solo puede escribir si no hay lectores.
En la red de Petri de un Mutex con dos procesos, ¿qué representa el lugar P2 inicializado con 1 token?
  • Que el proceso 2 está en ejecución
  • El recurso compartido disponible (el "lock"). El token se consume cuando un proceso entra a la sección crítica y se devuelve al salir
  • La cantidad de procesos que pueden entrar simultáneamente
  • El estado inicial del sistema que luego se descarta
En el modelo de mutex con Petri, el lugar compartido con 1 token actúa como semáforo binario. Solo uno de los procesos puede "tomar" ese token para entrar a la sección crítica. El otro debe esperar hasta que sea devuelto.
¿Qué información proporciona el Grafo de Alcance de una Red de Petri?
  • El código fuente equivalente en pseudocódigo
  • Todos los marcados (estados) que puede alcanzar la red desde el marcado inicial, y las transiciones que los conectan — permite detectar deadlocks
  • La performance teórica máxima del sistema modelado
  • El número mínimo de tokens necesarios para que la red funcione
El grafo de alcance es el grafo de estados del sistema. Si hay un estado sin transiciones salientes, ¡es un deadlock! También permite verificar propiedades como boundedness (¿el número de tokens en un lugar está acotado?) y liveness.
11 · Pasaje de Mensajes

Channels & Mensajes

En lugar de compartir memoria, los procesos se comunican enviando datos a través de canales tipados.

💬
Filosofía Go adoptada por Rust: "No comuniques compartiendo memoria; en cambio, compartí memoria comunicando." En vez de una variable compartida con locks, usá un canal para enviar el dato.

Modelos de Comunicación

Sincrónica

El emisor se bloquea hasta que el receptor recibe. Sin buffer. Garantiza que el receptor procesó antes de que el emisor continúe.

Asincrónica (buffered)

El emisor deposita en un buffer y continúa sin esperar. El receptor lo lee cuando puede.

Simétrico

Tanto el emisor como el receptor se conocen mutuamente. Ambos se nombran.

Unidireccional

Solo transporta datos en una dirección. Para bidireccional se usan dos canales.

MPSC en Rust — Multiple Producer, Single Consumer

use std::sync::mpsc;
use std::thread;

// tx = transmitter (sender), rx = receiver
let (tx, rx) = mpsc::channel();
let tx2 = tx.clone(); // múltiples productores

thread::spawn(move || {
    tx.send(String::from("Hola")).unwrap();
    // send transfiere ownership — no hay copia
});

// rx implementa Iterator — se cierra cuando todos los tx se dropean
for received in rx {
    println!("Recibido: {}", received);
}

Selective Input

Selective Input (select en Go)

Sintaxis que permite escuchar en varios canales de forma bloqueante y desbloquearse con el primero que recibe un mensaje. Fundamental para que un proceso pueda reaccionar a distintas fuentes de datos.

Canales en Unix: Pipes, FIFOs y Message Queues

  • Pipes: conectan dos procesos independientes, orientados a bytes, sin nombre
  • FIFOs: como pipes pero con representación en el filesystem (tienen nombre)
  • Message Queues: orientadas a mensajes como unidades independientes

Filósofos con Canales — Chandy/Misra

✓
Idea clave: Los palitos circulan por mensajes. Cada palito puede estar sucio o limpio. Un filósofo entrega un palito sucio si le es solicitado, pero retiene uno limpio. Esto garantiza ausencia de deadlock y starvation sin memoria compartida.

🎯 Quiz — Unidad 11

¿Cuál es la diferencia entre comunicación sincrónica y asincrónica por canales?
  • En sincrónica el mensaje llega instantáneamente, en asincrónica hay latencia
  • Sincrónica: el emisor se bloquea hasta que el receptor recibe (rendezvous). Asincrónica (buffered): el emisor deposita en un buffer y continúa sin esperar
  • Asincrónica garantiza que los mensajes llegan en orden, sincrónica no
  • Sincrónica requiere que emisor y receptor estén en el mismo proceso
Sincrónica = rendezvous: ambos se encuentran en el punto de comunicación, el emisor espera al receptor. Asincrónica = buzón: el emisor deja el mensaje y sigue; el receptor lo lee cuando puede. El canal de Rust (mpsc) es asincrónico por defecto.
¿Qué significa MPSC en el canal de Rust?
  • Multiple Processes, Single Channel
  • Multiple Producer, Single Consumer — múltiples hilos pueden enviar por el canal, pero solo uno puede recibir
  • Message Passing Synchronization Control
  • Mutable Protected Shared Channel
mpsc::channel() crea un par (tx, rx). Se pueden clonar múltiples tx (transmitters) para que múltiples threads envíen. Solo puede haber un rx (receiver). El canal se cierra automáticamente cuando todos los tx se dropean.
¿Qué ventaja de seguridad tiene usar canales en lugar de memoria compartida?
  • Los canales son más rápidos que los locks en todos los casos
  • Al transferir ownership del dato por el canal, no puede haber acceso concurrente — el emisor ya no tiene el dato cuando el receptor lo tiene
  • Los canales eliminan la necesidad de sincronización completamente
  • Los canales en Rust funcionan en sistemas distribuidos automáticamente
"Share memory by communicating": el send() transfiere ownership. El emisor ya no puede acceder al valor. No hay estado compartido → no hay data races posibles. Esto es lo que hace al modelo de canales inherentemente más seguro.
¿Para qué sirve Selective Input (select en Go)?
  • Para enviar el mismo mensaje a múltiples canales simultáneamente
  • Para escuchar en múltiples canales de forma bloqueante y desbloquearse con el primero que recibe un mensaje
  • Para filtrar mensajes por tipo antes de recibirlos
  • Para garantizar que los mensajes se procesan en orden FIFO
Selective Input es fundamental para procesos que deben reaccionar a múltiples fuentes. Sin select, habría que hacer polling en cada canal. Con select, el proceso se bloquea eficientemente y despierta cuando cualquiera de los canales tiene un mensaje.
En la solución de Chandy/Misra para filósofos con canales, ¿cuándo un filósofo entrega su palito?
  • Siempre que se lo pidan, inmediatamente
  • Solo si el palito está sucio — si está limpio lo retiene. Después de comer, los palitos quedan sucios
  • Solo después de comer, nunca antes
  • Cuando lleva más de N segundos con el palito
Regla Chandy/Misra: palito sucio → lo entrega limpiado. Palito limpio → lo retiene. Esto garantiza ausencia de deadlock y starvation: un filósofo que acaba de comer (palitos sucios) siempre cede a quien espera.
12 · Modelo de Actores

Actores & Actix

Modelo de concurrencia de Carl Hewitt (1973), popularizado por Erlang. Los actores encapsulan estado, comportamiento y tienen una dirección única.

📬
Mailbox (FIFO)
Cada actor tiene una cola de mensajes. Procesa uno a la vez. Sin acceso concurrente a su estado.
🔒
Estado Privado
No comparte memoria con otros actores. El estado solo cambia al procesar mensajes. Sin locks.
🌐
Distribuible
La dirección de un actor puede ser remota. El código no cambia entre local y remoto.
⚡
Livianos
Miles o millones de actores en lugar de threads del SO.

Framework Actix en Rust

// Definir el actor
struct MyActor;
impl Actor for MyActor {
    type Context = Context<Self>;
}

// Definir un mensaje
struct Ping;
impl Message for Ping {
    type Result = bool;
}

// Implementar el handler
impl Handler<Ping> for MyActor {
    type Result = bool;
    fn handle(&mut self, _msg: Ping, _ctx: &mut Context<Self>) -> bool {
        true
    }
}

Formas de enviar mensajes (Actix)

MétodoComportamiento
Addr::do_send(M)Fire-and-forget. Ignora errores. No retorna resultado.
Addr::try_send(M)Envío inmediato. Retorna SendError si mailbox lleno o cerrado.
Addr::send(M)Retorna un Future con el resultado. Permite .await.

Ciclo de vida del actor

Started

Se llama a started(). El contexto está disponible.

Running

Estado normal, indefinido. Procesa mensajes de la mailbox.

Stopping

Llamó a Context::stop, nadie lo referencia, o no hay objetos registrados.

Stopped

Estado final. No puede volver a un estado anterior.

⚠
Trampa común: No uses acquire() bloqueante dentro de un handler de actor — viola el modelo. El actor no debe bloquearse. Usa try_acquire() y maneja el caso de fallo de otra manera (reencolar, notificar, etc.).

🎯 Quiz — Unidad 12

¿Por qué el estado interno de un actor NO necesita un Mutex para ser thread-safe?
  • Porque los actores son inmutables por diseño
  • Porque el actor procesa mensajes de a uno por vez — el modelo garantiza que nunca hay acceso concurrente a su estado; la protección es por diseño, no por lock
  • Porque Actix implementa un Mutex interno automáticamente
  • Porque los actores solo pueden tener estado de tipos primitivos
Un actor tiene una mailbox (FIFO) y procesa un mensaje a la vez. Nunca hay dos threads ejecutando el handler del mismo actor simultáneamente. No hay concurrencia en el estado del actor — la seguridad viene del modelo, no de locks explícitos.
¿Cuál es la trampa de usar acquire() bloqueante dentro de un handler de actor?
  • Acquire no existe en el contexto de actores
  • Bloquea el thread del actor, impidiendo que procese otros mensajes — viola el modelo de actores y puede causar deadlocks
  • Solo es problema si el semáforo tiene valor 0
  • No es problema si el actor no tiene más mensajes en la mailbox
Los actores deben ser no-bloqueantes. Si un handler hace acquire() bloqueante, el thread queda congelado y ningún otro mensaje puede procesarse en ese Arbiter. Usar try_acquire() y manejar el fallo, o rediseñar con mensajes de request/response.
¿Cuál es la diferencia entre do_send, try_send y send en Actix?
  • Solo difieren en el nombre, el comportamiento es idéntico
  • do_send: fire-and-forget, ignora errores. try_send: retorna error si mailbox llena/cerrada. send: retorna Future con el resultado del handler
  • try_send es el más seguro porque garantiza entrega
  • send bloquea el thread emisor hasta que el receptor procesa el mensaje
do_send: máximo desacoplamiento, sin feedback. try_send: detecta errores de envío sin esperar resultado. send: permite await del resultado, útil cuando necesitás la respuesta del actor. Elegir según si necesitás confirmación/resultado.
¿En qué se diferencia el modelo de Actores del pasaje de mensajes con canales?
  • Los actores no pueden comunicarse entre sí, solo con el exterior
  • Los actores encapsulan estado privado y comportamiento como una unidad; pueden crear otros actores, tienen ciclo de vida propio, y su "canal" (mailbox) está integrado en la entidad
  • Los canales son más seguros porque usan ownership de Rust
  • Son equivalentes, actores son solo canales con mejor API
Canales son mecanismos de comunicación. Actores son entidades con identidad, estado, comportamiento y mailbox. Un actor puede crear hijos, supervisarlos, cambiar su comportamiento y tiene un ciclo de vida explícito. Son una abstracción de más alto nivel.
¿Cuándo un actor pasa al estado "Stopping" en Actix?
  • Cuando su mailbox está vacía por más de 5 segundos
  • Cuando llama Context::stop, cuando ningún otro actor lo referencia, o cuando no tiene objetos registrados en el contexto
  • Automáticamente después de procesar 1000 mensajes
  • Solo cuando el sistema de actores completo se detiene
Stopping ocurre por: detención explícita (ctx.stop()), garbage collection (nadie tiene su dirección), o falta de trabajo registrado. Desde Stopping puede volver a Running si llega un mensaje. Stopped es el estado final e irreversible.
