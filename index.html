01 · Introducción

Programación Concurrente

Fundamentos teóricos, vocabulario esencial y los cinco modelos de concurrencia estudiados en la materia.

Definiciones Clave

Antes de hablar de concurrencia, es fundamental entender qué es un programa secuencial y cómo se diferencia de uno concurrente.

Programa

Conjunto de datos, asignaciones e instrucciones de control de flujo que compilan a instrucciones de máquina. Se ejecutan secuencialmente en un procesador y acceden a datos almacenados en memoria principal o secundaria.

Programa Concurrente

Conjunto finito de procesos secuenciales que pueden ejecutarse en paralelo. Su ejecución resulta del intercalado arbitrario de instrucciones atómicas de cada proceso. No es tan simple como "corren al mismo tiempo": el resultado depende del orden en que se intercalen las instrucciones.

Proceso

Cada uno de los programas secuenciales que conforman el programa concurrente. Están compuestos por un conjunto finito de instrucciones atómicas.

Instrucción Atómica

Una instrucción que puede ejecutarse de principio a fin sin interrupciones, o directamente no ejecutarse. No hay estado intermedio observable. Ejemplo: un incremento x++ a nivel de C parece atómico, pero en assembler son tres instrucciones separadas (load, add, store).

Sistema Paralelo

Varios programas ejecutándose simultáneamente en procesadores físicos distintos. Paralelismo real.

Multitasking

Ejecución de múltiples procesos en un periodo de tiempo. El scheduler del kernel coordina el acceso a los procesadores, dando la ilusión de paralelismo.

Multithreading

Construcción que permite ejecución concurrente de threads dentro del mismo proceso. Los threads comparten el espacio de memoria del proceso pero tienen su propio stack, PC y registros.

Ejecución Concurrente

Resulta de ejecutar una secuencia de instrucciones atómicas que se obtiene de intercalar arbitrariamente las instrucciones de los procesos. El resultado puede variar.

ℹ
Threads vs Procesos: Los procesos son independientes y tienen memoria separada. Los threads son "procesos ligeros" dentro del mismo proceso — comparten la memoria del proceso, pero cada uno tiene su propia información de estado (stack, Program Counter, registros). Esto es lo que los hace más eficientes pero también más riesgosos.

Desafíos de la Concurrencia

La programación concurrente introduce problemas que no existen en programación secuencial. Los principales son:

⏱
Sincronización
Coordinación temporal entre distintos procesos. ¿Cuándo puede el proceso B continuar si depende de que el proceso A terminó algo?
📡
Comunicación
Datos que los procesos necesitan compartir para cumplir la función del programa. ¿Cómo pasan información entre sí sin corromperla?
⚠️
Race Conditions
Cuando el resultado depende del orden de ejecución de los threads. El programa se comporta diferente cada vez que corre.
🔒
Deadlock
Dos o más procesos bloqueados mutuamente esperando recursos que el otro tiene. Nadie puede avanzar.

Los 5 Modelos de Concurrencia

Existen distintas formas de encarar el problema de la concurrencia. La materia estudia cinco modelos principales, cada uno con sus ventajas y casos de uso ideales:

1. Estado Mutable Compartido

Los procesos comparten memoria y se serializa el acceso con locks. Solo un proceso a la vez puede ejecutar ciertas regiones de código. Es el modelo más bajo nivel.

2. Paralelismo Fork-Join

El cómputo se divide en sub-tareas independientes que se ejecutan en paralelo y luego se combinan. Ideal para algoritmos tipo divide-and-conquer.

3. Canales / Mensajes

Los procesos se comunican enviando mensajes a través de canales tipados. No comparten memoria directamente. Más seguro y fácil de razonar.

4. Programación Asincrónica

Tareas livianas que se interleavan cooperativamente. El cambio de tarea solo ocurre en puntos específicos (await). Ideal para I/O-bound.

5. Actores

Entidades independientes con estado privado que solo se comunican por mensajes. Cada actor procesa mensajes de a uno, sin compartir estado.

Serializar el Acceso

En el modelo de estado compartido: marcar regiones de código que no pueden superponerse en la ejecución. Si un proceso está en la región, los demás esperan.

02 · Lenguaje Base

Introducción a Rust

Rust es el lenguaje base de la materia. Sus tres pilares son velocidad, seguridad y concurrencia. Está diseñado para que el compilador detecte errores que en C/C++ solo aparecen en runtime.

🚀
Velocidad
Zero-cost abstractions: pagás solo por lo que usás. Performance comparable a C/C++ sin garbage collector.
🛡
Memory Safety
Sin punteros sueltos (dangling pointers), sin buffer overflow, sin doble liberación. Todo se verifica en tiempo de compilación.
⚙️
Concurrencia
El compilador previene data races. Si compila, no hay race conditions sobre los datos.
📦
Cargo
Herramienta de build integrada + gestor de dependencias (crates.io). Un solo comando para compilar, testear, formatear.

Tipos de Datos

Rust es fuertemente tipado. El compilador puede inferir tipos en muchos casos, pero siempre son verificados en compile-time.

Enteros con signo

i8, i16, i32, i64 — El número indica los bits. isize depende de la arquitectura del target.

Enteros sin signo

u8, u16, u32, u64 — Solo valores positivos. usize se usa para índices y tamaños.

Flotantes y bool

f32, f64 — Punto flotante. bool ocupa 1 byte. char es Unicode (4 bytes).

Compuestos

Arrays: [u8; 3]. Slices: arr[3..5]. Tuplas: (char, u8, i32). Dinámicos: Vec<T>, String.

Variables e Inmutabilidad

Una de las decisiones más importantes de Rust: las variables son inmutables por defecto. Hay que pedir explícitamente mutabilidad. Esto fuerza al programador a pensar en qué cosas deben cambiar.

rust// Las variables son inmutables por default
let t = true;
// t = false; // ERROR: no se puede asignar a variable inmutable

// Para hacerlas mutables usamos mut
let mut punto = (1u8, 2u8);
punto = (4, 3);  // OK, es mutable

// Los tipos pueden ser inferidos o explícitos. La asignación = "binding"
let numero: i32 = 42;

Funciones y Closures

Las funciones en Rust usan la sintaxis fn nombre(param: tipo) -> tipo_retorno. El último valor sin punto y coma es el valor de retorno implícito.

rust// Función normal
fn sumar_uno(a: i32) -> i32 {
    a + 1  // sin punto y coma = retorno implícito
}

// Closure (función anónima, captura el entorno)
let plus_one = |a| { a + 1 };

// Función que recibe otra función como parámetro
fn map42(mapper: fn(i32) -> i32) -> i32 {
    mapper(42)
}

Structs y sus Métodos

Los structs agrupan datos relacionados. Con impl se definen los métodos. El primer parámetro &self indica que es un método de instancia (como this en otros lenguajes).

rust// Struct con campos con nombre
#[derive(Debug)]
struct Persona {
    nombre: String,
    apellido: String,
}

// Struct posicional (tuple struct)
struct NumeroImaginario(f64, f64);

// Métodos del struct
impl NumeroImaginario {
    // "Método estático" / constructor (no recibe &self)
    fn new(r: f64, i: f64) -> NumeroImaginario {
        NumeroImaginario(r, i)
    }

    // Método de instancia (recibe &self, referencia al objeto)
    fn modulo(&self) -> f64 {
        (self.0 * self.0 + self.1 * self.1).sqrt()
    }
}

Traits: interfaces de Rust

Los traits son como las interfaces en otros lenguajes. Definen un conjunto de métodos que un tipo debe implementar. Permiten el polimorfismo en Rust.

rust// Definir un trait
trait MagnitudVectorial {
    fn norma(&self) -> f64;
}

// Implementar el trait para nuestro tipo
impl MagnitudVectorial for NumeroImaginario {
    fn norma(&self) -> f64 {
        self.modulo()
    }
}

// Función polimórfica: acepta cualquier tipo que implemente el trait
fn max(v1: &dyn MagnitudVectorial, v2: &dyn MagnitudVectorial) -> f64 {
    v1.norma().max(v2.norma())
}

Enums y Pattern Matching

Los enums en Rust son mucho más poderosos que en otros lenguajes: cada variante puede contener datos. El pattern matching con match obliga a manejar todos los casos.

rustenum Message {
    Fire,
    Move { x: i32, y: i32 },  // con campos nombrados
    Say(String),               // con dato
}

fn print_message(m: Message) {
    match m {
        Message::Fire => println!("Fire"),
        // Guard adicional: solo si y > 10
        Message::Move { x: _, y } if y > 10 => println!("Corre hacia arriba"),
        Message::Move { x, y } => println!("Se mueve hacia {}, {}", x, y),
        Message::Say(msg) => println!("Dice: {}", msg),
    };
}

Option y Result: manejo de errores sin excepciones

Rust no tiene null ni excepciones. En su lugar usa tipos que fuerzan al programador a manejar los casos de ausencia de valor y error explícitamente.

rust// Option representa un valor que puede existir o no
fn dividir(num: f64, den: f64) -> Option<f64> {
    if den == 0.0 { None } else { Some(num / den) }
}

// Result representa éxito o error con información
fn abrir_archivo(path: String) -> Result<String, String> {
    // ... lógica ...
    Err(String::from("no existe el archivo"))
}

let d = dividir(3.0, 0.0);
d.unwrap() * 3;            // PANIC si es None
d.expect("dividió por 0");   // PANIC con mensaje
d.map(|n| n * 3);          // Transforma el valor si existe, sin panic

// El operador ? propaga el error automáticamente
fn div_mul(n: f64, d: f64, m: f64) -> Option<f64> {
    let d = dividir(n, d)?;  // si es None, retorna None aquí
    Some(d * m)
}

02b · Rust Avanzado

Ownership & Borrowing

El sistema de ownership es lo que hace a Rust único. Permite garantizar memory safety sin garbage collector, eliminando clases enteras de errores en tiempo de compilación.

Motivación: ¿Qué problemas resuelve?

El sistema de ownership está inspirado en C++ RAII (Resource Acquisition Is Initialization) pero lo lleva al extremo. El objetivo es eliminar en compile-time todos estos bugs:

  • Punteros nulos — acceder a memoria que ya fue liberada
  • Uso posterior a liberación (use-after-free) — usar un puntero después de liberar la memoria
  • Doble liberación (double-free) — liberar la misma memoria dos veces
  • Buffer overflow — escribir más allá del final de un array
  • Data races — dos threads acceden a la misma memoria concurrentemente sin sincronización
⚠
Costo cero en runtime: Todas estas garantías se verifican en tiempo de compilación. El programa compilado no tiene overhead por estas verificaciones — son cero costo en ejecución.

Las 3 Reglas de Ownership

1
Cada valor tiene un único dueño (owner)

Solo una variable puede ser el owner de un valor en un momento dado. No puede haber dos owners del mismo valor.

2
Cambiar de dueño es un move

Al asignar a otra variable o pasar a una función, la variable original queda inválida. Ya no se puede usar.

3
Cuando el owner sale de scope, el valor se elimina

Se llama automáticamente al trait Drop. Similar a los destructores de C++. No hace falta free() manual.

Move vs Borrow

Si una función necesita el valor solo para leerlo o modificarlo temporalmente, no tiene sentido transferir el ownership. Para eso existe el borrowing: prestar una referencia.

Reglas de Borrowing

Múltiples referencias inmutables

El dueño puede prestar muchas referencias de lectura simultáneamente. Todos pueden leer al mismo tiempo, es seguro.

Una única referencia mutable

Solo puede existir UNA referencia mutable a la vez. Mientras existe, no puede haber referencias inmutables. Esto elimina data races.

// ✓ Múltiples referencias inmutables OK
let r1 = &ariel;
let r2 = &ariel;
println!("{:?} {:?}", r1, r2);

// ✓ Una referencia mutable
let r3 = &mut ariel;
r3.edad += 1;

// ✗ No se puede tener mutable + inmutable al mismo tiempo
// let r4 = &ariel; // ERROR si r3 sigue activo

Heap y Punteros Inteligentes

Para datos de tamaño dinámico o para compartir datos entre múltiples owners, Rust provee punteros inteligentes:

Box<T>
Puntero a heap con un único owner. Box<str> → String, Box<[T]> → Vec<T>
Rc<T>
Reference Counting. Múltiples owners inmutables. Solo para un solo thread.
Arc<T>
Atomic Reference Counting. Como Rc pero thread-safe. Para compartir entre múltiples threads.
Arc<RwLock<T>>
Patrón típico: compartir entre threads con acceso mutable controlado por un lock.

Traits Send y Sync

Rust garantiza thread-safety en compile-time a través de dos traits especiales que el compilador verifica automáticamente:

Send

El ownership del tipo puede ser transferido entre threads. Casi todos los tipos son Send. Excepción notable: Rc<T> (no es atómico). Los tipos compuestos de tipos Send son automáticamente Send.

Sync

Es seguro ser referenciado desde múltiples threads simultáneamente. T es Sync si &T es Send. Los tipos primitivos son Sync. Los tipos compuestos de tipos Sync son automáticamente Sync.

Threads en Rust

rustuse std::{thread, time::Duration};

println!("[PADRE] spawneo hijo");

// thread::spawn crea un nuevo thread
// 'move' indica que el closure toma ownership de las variables capturadas
let join_handle = thread::spawn(move || {
    println!("[HIJO] spawnie");
    thread::sleep(Duration::from_secs(2));
    println!("[HIJO] me desperté");
});

println!("[PADRE] esperando hijo");
join_handle.join().unwrap();  // bloquea hasta que el hijo termina
println!("[PADRE] terminó");
✓
¿Por qué move? El thread hijo puede vivir más que el scope del padre. Si el closure capturara referencias del padre en vez de tomar ownership, esas referencias podrían quedar inválidas. move fuerza al closure a ser dueño de todo lo que usa.
03 · Modelo de Paralelismo

Fork-Join

Modelo de paralelismo cuyo origen se remonta al paper de Melvin Conway en 1963. El cómputo se divide recursivamente en sub-tareas independientes que se ejecutan en paralelo y luego se combinan.

Procesamiento Secuencial vs Paralelo

Para entender Fork-Join, primero veamos el procesamiento secuencial clásico y su limitación:

rust// Versión secuencial: los archivos se procesan uno por uno
fn process_files(filenames: Vec<String>) -> io::Result<()> {
    for document in filenames {
        let text = load(&document)?;     // leer un archivo
        let results = process(text);    // realizar el cómputo
        save(&document, results)?;       // escribir resultados
    }
    Ok(())
}

Si tenemos 8 núcleos y 8 archivos, el procesamiento secuencial usa solo 1 núcleo. Desperdicia el 87.5% del hardware disponible.

🔀
División Recursiva
El cómputo se parte recursivamente. Los sub-cómputos son independientes entre sí → pueden ejecutarse en paralelo sin coordinación.
🎲
Determinístico
Los threads están aislados. El programa produce el mismo resultado independientemente de las diferencias de velocidad entre threads.
⚡
Performance Ideal
En el caso ideal: t_secuencial / N_threads. Puede variar por diferencias en tamaño de tareas y costo de combinar resultados.
⚠️
Desventaja
Requiere que las unidades de trabajo (tareas) sean aisladas entre sí. No funciona si las tareas comparten estado mutable.

Reglas del Modelo Fork-Join

  • Las sub-tareas se pueden crear en cualquier momento de la ejecución de la tarea padre.
  • Las tareas no deben bloquearse, excepto para esperar el final de sus subtareas (join).
  • Es un modelo de concurrencia sin condiciones de carrera (por el aislamiento de tareas).

Work Stealing: el corazón del scheduling

En Fork-Join, las tareas pueden tener tamaños muy diferentes. Si las asignamos estáticamente a los threads, algunos terminarán antes y estarán ociosos mientras otros siguen trabajando. El algoritmo Work Stealing resuelve esto dinámicamente:

1
Cada thread tiene su propia cola de doble extremo (deque)

Almacena las tareas listas para ejecutar. Cada thread trabaja principalmente con su propia cola.

2
Al terminar una tarea, encola las subtareas al final

Las subtareas creadas se agregan al final de la propia cola del thread.

3
Toma la siguiente tarea del final de su propia cola

LIFO local: favorece la localidad de caché, las subtareas recién creadas están calientes.

4
Si la cola está vacía: roba del inicio de una cola aleatoria

FIFO en el robo: minimiza contención. Los threads inactivos roban a los ocupados para balancear la carga.

✓
Ventajas de Work Stealing: Los worker threads solo se comunican cuando lo necesitan → menor necesidad de sincronización. La implementación de la cola deque agrega bajo overhead de sincronización.

Implementación con la Standard Library

rustfn process_files_in_parallel(filenames: Vec<String>) -> io::Result<()> {
    const NTHREADS: usize = 8;
    let worklists = split_vec_into_chunks(filenames, NTHREADS);
    let mut thread_handles = vec![];
    
    for worklist in worklists {
        thread_handles.push(
            std::thread::spawn(move || process_files(worklist))
        );
    }
    
    for handle in thread_handles {
        handle.join().unwrap()?;  // esperar a todos los threads
    }
    Ok(())
}

Crossbeam: scoped threads

Crossbeam es un crate de concurrencia muy utilizado. crossbeam::scope resuelve un problema importante: los threads de std::thread no pueden tomar referencias al stack del padre (porque el padre podría terminar antes). Crossbeam garantiza que todos los threads del scope terminan antes de que el scope cierre.

rustuse crossbeam;

let data = vec![1, 2, 3];

crossbeam::scope(|s| {
    // Podemos tomar referencias al stack porque scope garantiza que
    // todos los threads terminan antes de que scope retorne
    s.spawn(|_| {
        println!("data = {:?}", data);  // referencia a 'data', no move
    });
}).unwrap();
// Aquí todos los threads terminaron. Ok o Err si alguno hizo panic.
04 · Paradigmas de Datos

MapReduce & Dremel

Paradigmas de procesamiento masivo de datos que se construyen sobre los conceptos de Fork-Join. MapReduce (2004) popularizó el procesamiento distribuido de grandes datasets.

MapReduce

Presentado en un paper de Google en 2004, MapReduce es un modelo de programación para procesar y generar grandes datasets. El usuario especifica dos funciones:

Función Map

Toma un par clave/valor y genera un conjunto de pares clave/valor intermedios. Se ejecuta en paralelo sobre distintas particiones de datos.

Función Reduce

Combina todos los valores intermedios asociados con la misma clave. Produce el resultado final.

Ejemplo: Word Count (el "Hello World" de MapReduce)

M
Map — Extraer frecuencias locales

Tomar de potencialmente varios archivos de texto las líneas, de las líneas las palabras, de las palabras sus frecuencias locales (cuántas veces aparece cada palabra en este fragmento).

R
Reduce — Combinar frecuencias globales

Combinar todas las frecuencias locales en una única tabla con todas las frecuencias globales.

ℹ
Ecosistema: Hadoop fue la primera implementación open source de MapReduce. Sus "hijos" modernos incluyen Apache Spark, que mejoró sustancialmente la performance al mantener datos en memoria entre operaciones.

Dremel

Sistema de consultas en tiempo casi real, expuesto públicamente como BigQuery de Google. La idea central es distribuir el trabajo en un árbol de nodos:

Arquitectura en árbol

Una consulta se propaga hacia abajo en el árbol y se reescribe en cada nivel. El resultado se ensambla agregando las respuestas que llegan desde los niveles inferiores. Permite ejecutar queries analíticas sobre petabytes de datos en segundos.

Vectorización

Cuando queremos un cómputo "simple" sobre un conjunto grande de datos (procesar sonido, imágenes, video, entrenar redes neuronales), el overhead de Fork-Join puede ser excesivo. Lanzar incluso una tarea tiene un costo en cómputo y accesos a memoria (localidad). Para esto existen las operaciones vectoriales.

05 · Paralelismo de Datos

Vectorización & SIMD

Al "detenerse" la ley de Moore, el aumento de transistores no se tradujo en mayor velocidad de reloj. En su lugar se agregaron múltiples ALUs para operar sobre los mismos registros de forma concurrente.

SIMD — Single Instruction, Multiple Data

Instrucciones de CPU que operan sobre vectores de datos en paralelo. Introducido con los juegos de instrucciones MMX, SSE, AVX en x86 y NEON en ARM. Cada variable (registro) es un vector de tamaño fijo, generalmente entre 128 y 512 bits, dividido en N "lanes" según el tamaño de los datos.

Tipos de Operaciones

Operaciones "Verticales"

Entre distintos registros en el mismo lane. Son las más rápidas. Ejemplo: sumar todos los x₀ con todos los y₀ simultáneamente.

Operaciones "Horizontales"

Reducen un vector a un escalar. Son más lentas porque requieren combinar datos de distintos lanes. Ejemplo: sumar todos los elementos de un registro.

Ejemplo: un registro de 512 bits

Un registro de 512 bits puede alojar simultáneamente:

  • 64 enteros de 8 bits (u8/i8)
  • 16 enteros de 32 bits (u32/i32)
  • 8 flotantes de 64 bits (f64)

Con una sola instrucción SIMD se pueden sumar, multiplicar o comparar todos esos valores en paralelo. Esto es lo que hace posible el entrenamiento eficiente de redes neuronales.

GPU vs CPU

CPU — Low Latency
  • Diseñada para minimizar la latencia
  • Caches grandes (L1, L2, L3)
  • Lógica complicada (Pipeline, branch prediction)
  • Pocos núcleos potentes
  • Context switch caro (~microsegundos)
GPU — High Throughput
  • Diseñada para maximizar el throughput
  • Más transistores en ALUs, caches pequeños
  • Miles de núcleos simples
  • Los tiempos de espera de RAM se diluyen intercalando threads
  • Context switch muy económico (hasta uno por clock cycle)

Modelo de Programación GPU: Jerarquía de Threads

El programa se divide en un grid de workgroups independientes. Cada workgroup tiene N threads que se ejecutan en paralelo sobre el mismo espacio de memoria compartida — de forma equivalente a un lane SIMD. La performance óptima se logra cuando la cantidad de threads en un workgroup es múltiplo de 32 (el tamaño del "warp" en CUDA o "SIMD-group" en Metal).

WGSL — WebGPU Shader Language

Lenguaje de shaders moderno, portable entre Vulkan, Metal y Direct3D 12. Con sintaxis similar a Rust (strongly typed). Permite escribir programas que corren en la GPU de manera portable.

Flujo típico de un programa GPU

1
Declarar y allocar memoria en host y device

La GPU tiene su propia VRAM separada de la RAM del CPU.

2
Inicializar datos en el host

Preparar los datos en la RAM del CPU.

3
Transferir datos host → device

Copiar los datos a la VRAM (a través de zonas de memoria "mappeable").

4
Ejecutar shaders en la GPU

Lanzar uno o más shaders (funciones que corren en la GPU). Esperar a que terminen.

5
Transferir resultados device → host

Copiar los resultados de vuelta a la RAM para usarlos en el CPU.

06 · Modelo Asincrónico

Async / Await & Futures

Modelo de concurrencia basado en la cooperación entre tareas livianas. Ideal para operaciones I/O-bound donde los threads pasarían la mayor parte del tiempo esperando system calls sin hacer trabajo real.

El Problema de los Threads para I/O

Si una aplicación crea muchos threads (ej: uno por conexión HTTP), cada thread puede necesitar 100KB de stack o más. Con 10.000 conexiones simultáneas, eso son 1GB solo de stacks. Además, la mayor parte del tiempo los threads están bloqueados esperando que el sistema operativo complete una operación de red o disco.

❌ No usar async cuando:

Calcular un factorial, producto de matrices, el problema de los filósofos, estado interno compartido. Son tareas CPU-bound: el tiempo lo pasa computando, no esperando. Usá Fork-Join.

✓ Usar async cuando:

Consultar servicios externos, leer archivos, servir requests HTTP. Son tareas I/O-bound: el tiempo lo pasa esperando respuestas. El thread podría atender otras tareas mientras tanto.

Tareas Asincrónicas vs Threads

Creación~15 microsegundos~300 nanosegundos (50x más rápido)
Context switchCaro (syscall al kernel)Más rápido (cooperativo, en userspace)

Futures: el concepto central

Rust introduce el trait std::future::Future para representar operaciones asincrónicas. Un Future es un valor que representa un cómputo que puede no estar completo todavía.

rust// Definición simplificada del trait Future
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

// Poll puede ser Ready (con el valor) o Pending (todavía no terminó)
enum Poll<T> {
    Ready(T),
    Pending,
}
💡
La metáfora de la piñata: Lo único que se puede hacer con un Future es "golpearlo con poll" hasta que caiga el valor. El método poll nunca bloquea: si la operación no está lista, retorna Pending inmediatamente. Si terminó, retorna Ready(output).

Cómo funciona poll en la práctica

1°
Primera llamada a poll — el future inicia la operación (ej: envía el request HTTP) y se registra para ser notificado.
Pending
2°
Segunda llamada a poll — la operación todavía no terminó (el servidor no respondió aún).
Pending
3°
Tercera llamada a poll — la respuesta llegó. El future retorna el resultado final.
Ready(T)

La arquitectura async de Rust está diseñada para ser eficiente: se llama a poll solo cuando vale la pena (algo debe retornar Ready, o progresar hacia el objetivo). El OS provee system calls como interfaz para hacer poll sobre operaciones de I/O.

Funciones Async y el operador await

Escribir código que usa poll directamente sería muy tedioso. Rust provee las keywords async y await que generan automáticamente el código de polling:

rust// Versión sincrónica (bloquea el thread mientras espera)
fn cheapo_request_sync(host: &str, port: u16, path: &str) -> io::Result<String> {
    let mut socket = net::TcpStream::connect((host, port))?;
    // ... todo el thread se bloquea aquí esperando la conexión
    socket.read_to_string(&mut response)?;  // y aquí esperando la respuesta
    Ok(response)
}

// Versión asincrónica (libera el thread mientras espera)
async fn cheapo_request(host: &str, port: u16, path: &str) -> io::Result<String> {
    let mut socket = net::TcpStream::connect((host, port)).await?;
    // .await suspende ESTA tarea pero libera el thread para otras tareas
    socket.read_to_string(&mut response).await?;
    Ok(response)
}

¿Qué hace exactamente async/await?

1
Invocar una función async retorna inmediatamente

Antes de que comience a ejecutarse el cuerpo. Se obtiene un Future que contiene todo lo necesario para ejecutarla.

2
Al ejecutar poll por primera vez

Se ejecuta el cuerpo de la función hasta el primer await. Cada await es un punto potencial de suspensión.

3
Si el sub-future no está listo: Pending

Toda la función retorna Pending y el thread queda libre para ejecutar otras tareas.

4
Próximo poll: continúa desde donde se suspendió

El Future almacena el estado local y el punto de reanudación. Continúa desde el await anterior.

⚠
Diferencia clave con threads: El cambio de una tarea a otra ocurre SOLO en las expresiones await. Un cómputo CPU-intensivo en una función async bloquearía al thread y no daría lugar a otras tareas. Para cómputos pesados dentro de async, usar async_std::task::spawn_blocking que mueve la tarea a un thread del OS dedicado a tareas bloqueantes.

block_on: el puente entre async y sync

rustfn main() -> io::Result<()> {
    use async_std::task;
    
    // block_on: función SINCRÓNICA que ejecuta un Future
    // bloquea el thread actual hasta que el Future retorna Ready
    // No debe llamarse dentro de una función async
    let response = task::block_on(cheapo_request("example.com", 80, "/"))?;
    println!("{}", response);
    Ok(())
}

Crear tareas async: spawn

rustuse async_std::task;

// spawn_local: agrega al pool de la tarea actual (single-thread)
task::spawn_local(async {
    do_something().await;
});

// spawn: crea tarea en el pool de threads (multi-thread)
// Los lifetimes deben ser 'static (vive hasta el final del programa)
task::spawn(async {
    do_something().await;
});

// yield_now: favorece el paralelismo — cede el control a otra tarea
task::yield_now().await;  // Pending la primera vez, Ready(()) la segunda

Encadenar Futures

Los futures se pueden encadenar con await (secuencial) o con join! (paralelo):

rust// Secuencial: hello termina ANTES de que world empiece
async fn async_main_seq() -> String {
    hello().await + &world().await
}

// Paralelo con join!: hello y world se ejecutan concurrentemente
async fn async_main_par() -> String {
    let (h, w) = futures::join!(hello(), world());
    h + &w
}

Runtimes Async

Los runtimes como Tokio y async-std son los que realmente ejecutan los futures. Incluyen executors (que traducen await en llamadas a poll), bibliotecas async de I/O, timers, locks, channels y más.

ℹ
Pin<T>: Los tipos autogenerados de async que implementan Future guardan referencias a sí mismos (self-references). Si el future fuera movido en memoria, esas referencias internas quedarían inválidas. Pin "fija" el future en una dirección de memoria, haciendo imposible moverlo si implementa !Unpin.
07 · Sincronización

Corrección & Locks

En programas secuenciales, ante la misma entrada siempre se obtiene la misma salida. En programas concurrentes, la salida puede depender del escenario de ejecución — el orden en que se intercalan las instrucciones.

Propiedades de Corrección

Para razonar sobre si un programa concurrente es correcto, se usan dos categorías de propiedades:

Safety — debe ser verdadera SIEMPRE
Exclusión Mutua Dos procesos no deben intercalar ciertas (sub)secuencias de instrucciones. Ejemplo: el incremento de una variable global debe ser atómico.
Ausencia de Deadlock Un sistema que aún no finalizó debe poder continuar realizando su tarea, es decir, avanzar productivamente.
Liveness — debe volverse verdadera EVENTUALMENTE
Ausencia de Starvation Todo proceso que esté listo para utilizar un recurso debe recibir dicho recurso eventualmente. Ningún proceso espera para siempre.
Fairness (equidad) Un escenario es (débilmente) fair si en algún estado del escenario, una instrucción que está continuamente habilitada, eventualmente aparece en el escenario.

La Sección Crítica

El problema central de la sincronización. Cada proceso tiene código que puede dividirse en parte crítica (donde accede a recursos compartidos) y parte no-crítica.

Sección Crítica

Código que accede a recursos compartidos. Debe progresar (finalizar eventualmente). Solo un proceso a la vez puede ejecutarla.

Sección No-Crítica

El resto del código. No requiere progreso: el proceso puede terminar o entrar en un loop infinito aquí sin afectar a los demás.

Las especificaciones de corrección para la sección crítica son:

  • Exclusión mutua: no deben intercalarse instrucciones de la sección crítica de distintos procesos.
  • Ausencia de deadlock: si dos procesos están tratando de entrar, eventualmente alguno debe tener éxito.
  • Ausencia de starvation: si un proceso trata de entrar, eventualmente debe tener éxito.

Locks: la solución práctica

Los locks son el mecanismo más común para implementar exclusión mutua. Se implementan mediante variables de tipo lock que representan el estado del mismo, y requieren soporte tanto del hardware como del sistema operativo.

Lock

Variable con dos métodos: lock() — el proceso se bloquea hasta poder obtener el lock. unlock() — el proceso libera el lock que tomó previamente. Solo un proceso puede tener el lock a la vez.

Locks en UNIX

En UNIX, los locks fueron diseñados originalmente como mecanismo de sincronización de acceso a archivos, pero se pueden usar para cualquier recurso compartido. En Unix son advisory (los procesos pueden ignorarlos, es una convención).

Shared Lock (Read Lock)

Más de un proceso puede tener el lock al mismo tiempo. Se usa para proteger lecturas: múltiples lectores concurrentes son seguros.

Exclusive Lock (Write Lock)

Solo un proceso puede tener el lock. Excluye tanto a otros escritores como a lectores. Se usa para proteger escrituras.

Condiciones para tomar un lock

  • Para tomar un shared (read) lock: esperar hasta que sean liberados todos los exclusive locks.
  • Para tomar un exclusive (write) lock: esperar hasta que sean liberados todos los locks (de ambos tipos).

RwLock en Rust

Rust provee locks compartidos (lectura) y exclusivos (escritura) en std::sync::RwLock. Requiere que T sea Send para ser compartido entre threads y Sync para permitir acceso concurrente entre lectores.

rustuse std::sync::{Arc, RwLock};

let lock = RwLock::new(1);

// Lock de LECTURA: múltiples threads pueden leer simultáneamente
let n = lock.read().unwrap();
println!("El valor encontrado es: {}", *n);
// El lock de lectura se libera automáticamente cuando 'n' sale de scope (RAII)

// Lock de ESCRITURA: acceso exclusivo
let mut w = lock.write().unwrap();
*w += 1;
// El lock de escritura se libera automáticamente cuando 'w' sale de scope

// Compartir el RwLock entre threads: Arc (Atomic Reference Counter)
let shared = Arc::new(RwLock::new(0i32));
let shared_clone = shared.clone();  // incrementa el contador atómico
std::thread::spawn(move || {
    let mut w = shared_clone.write().unwrap();
    *w += 1;
});

Locks Envenenados (Poisoned Locks)

⚠
Lock Envenenado: Un lock queda en estado "poisoned" (envenenado) cuando un thread toma el lock de forma exclusiva y ejecuta panic!() mientras lo tiene. Las llamadas posteriores a read() y write() devolverán Err. Esto es intencional: el estado compartido puede haber quedado inconsistente, y Rust te obliga a manejarlo explícitamente.
08 · Sincronización II

Semáforos

Mecanismos de sincronismo de más alto nivel que los locks simples. Introducidos por Dijkstra en los años 60. Son un contador con dos operaciones atómicas.

Semáforo

Tipo de dato compuesto por dos campos: un entero no negativo V (el contador) y un conjunto de procesos L (los bloqueados esperando). Se inicializa con un valor k ≥ 0 y el conjunto vacío ∅. Dos operaciones atómicas: wait(S) también llamada p(S), y signal(S) también llamada v(S).

Un semáforo es esencialmente un contador de recursos disponibles:

  • Si contador > 0 → recurso disponible, se puede acceder
  • Si contador ≤ 0 → recurso no disponible, el proceso espera

Semántica de las Operaciones

wait(S) — p(S) — "tomar el recurso"
if S.V > 0:
    S.V := S.V - 1   // hay recurso: tomarlo
else:
    S.L.add(p)        // sin recurso: bloquear
    p.state := blocked

Decrementa el contador. Si llega a 0 o menos, el proceso se bloquea.

signal(S) — v(S) — "liberar el recurso"
if S.L.isEmpty():
    S.V := S.V + 1   // nadie espera: liberar
else:
    q := S.L.takeOne()  // despertar uno
    q.state := ready

Incrementa el contador. Si hay procesos esperando, despierta uno.

Propiedades Importantes

  • wait() y signal() son instrucciones atómicas (no se pueden interrumpir a la mitad).
  • Un semáforo debe inicializarse con un valor entero no negativo.
  • La instrucción signal() debe despertar a uno de los procesos suspendidos, pero no está definido cuál.

Invariantes de los Semáforos

Invariante 1

S.V ≥ 0 — El valor del semáforo nunca es negativo.

Invariante 2

S.V = k + #signal(S) − #wait(S) — siendo k el valor inicial. El valor refleja el balance de señales vs esperas.

Semáforo Binario (Mutex)

Si el valor V solo puede ser 0 o 1, se llama semáforo binario o Mutex. Se comporta igual que un lock de escritura exclusivo.

pseudocode// Inicializado en (1, ∅) — disponible
// Inicializado en (0, ∅) — ocupado desde el inicio

signal(S):
    if S.V = 1:
        // undefined (ya está libre, no debería señalizarse)
    else if S.L.isEmpty():
        S.V := 1         // nadie espera, simplemente liberar
    else:
        q := S.L.remove()   // despertar a uno
        q.state := ready

Semáforos en Rust

rustuse std_semaphore::Semaphore;

// Inicializar con 5 recursos disponibles
let sem = Semaphore::new(5);

// wait(): obtener acceso (decrementa el contador, bloquea si es 0)
sem.acquire();

// signal(): liberar acceso (incrementa el contador)
sem.release();

// Patrón RAII: libera automáticamente al salir del scope
let _guard = sem.access();  // se libera cuando _guard se dropea

Barreras en Rust

Las barreras son mecanismos de sincronización que permiten que varios threads esperen mutuamente hasta que todos llegaron a un punto determinado del algoritmo.

rustuse std::sync::{Arc, Barrier};

// Crear una barrera para sincronizar 3 threads
let barrier = Arc::new(Barrier::new(3));

// Cada thread llama a wait() cuando llega a la barrera
// Todos se bloquean hasta que los 3 llegaron
let bw = barrier.wait();

// is_leader() es true en exactamente un thread (el último en llegar)
// Útil para hacer alguna acción solo una vez cuando todos llegaron
if bw.is_leader() {
    println!("Todos llegaron! Soy el líder.");
}
✓
Las barreras son reutilizables automáticamente. Después de que todos los threads pasan, la barrera se reinicia y puede usarse para la próxima ronda de sincronización.

Problema Clásico: Productor-Consumidor

El problema del productor-consumidor modela una situación muy común: procesos que generan datos (productores) y procesos que consumen esos datos (consumidores), comunicándose a través de un buffer compartido.

Requisitos (invariantes que siempre deben cumplirse):

  1. No se puede consumir lo que no hay (buffer vacío)
  2. Todos los ítems producidos son eventualmente consumidos
  3. Al espacio de almacenamiento se accede de a uno (mutex)
  4. Se respeta el orden de almacenamiento y retiro (FIFO)

Caso 1: Buffer Infinito

Productor
loop forever:
  p1: append(d, buffer)
  p2: signal(notEmpty)

Produce dato → señaliza que ya hay algo

→
Consumidor
loop forever:
  q1: wait(notEmpty)
  q2: d ← take(buffer)

Espera a que haya algo → consume

Con buffer infinito solo usamos un semáforo notEmpty (0, ∅) — el problema solo es "no consumir si vacío".

Caso 2: Buffer Acotado (N lugares)

Se agregan dos problemas: no consumir si vacío y no producir si lleno. Necesitamos dos semáforos:

SemáforoValor inicialPropósito
notEmpty(0, ∅)Bloquea al consumidor si el buffer está vacío
notFull(N, ∅)Bloquea al productor si el buffer está lleno

Tipos de Semáforos en UNIX

System V

El semáforo guarda: el valor, el PID del último proceso que lo usó, la cantidad de procesos esperando, y la cantidad esperando que el semáforo sea cero.

POSIX

API más moderna y portátil. Más simple de usar. Dos variantes: con nombre (persistente en el filesystem) y sin nombre (para uso entre threads del mismo proceso).

09 · Sincronización III

Monitores & Condition Variables

Los monitores son una herramienta de sincronización de más alto nivel que los semáforos. Encapsulan el estado compartido junto con las operaciones que lo modifican, y garantizan exclusión mutua implícitamente.

¿Por qué necesitamos monitores?

Con semáforos, es fácil cometer errores: olvidar un signal, poner el wait en el lugar incorrecto, etc. Los monitores son más seguros porque la exclusión mutua es automática: el programador no tiene que preocuparse por lockear/unlockear manualmente.

Monitor

Herramienta de sincronización que permite a los hilos tener exclusión mutua y la posibilidad de esperar (block) a que una condición se cumpla. Tiene un mecanismo para señalizar a otros hilos cuando su condición se satisface. Solo un proceso puede ejecutar el monitor a la vez.

Estructura de un Monitor

Monitor
Variables internas
Estado compartido protegido por el monitor. Solo accesible desde los procedimientos del monitor.
Procedimientos del monitor
Rutinas que acceden directamente a las variables internas. Solo uno ejecuta a la vez.
Condition variables
Permiten que un proceso espere dentro del monitor liberando el lock temporalmente.
Interfaz pública
Los procesos acceden al estado solo a través de los procedimientos. Nunca directamente.

Estados de un proceso en un Monitor

  • Esperando para entrar al monitor — quiere llamar a un procedimiento pero otro proceso está dentro
  • Ejecutando el monitor — sólo un proceso a la vez (exclusión mutua)
  • Bloqueado en FIFO de condition variable — esperando que una condición se cumpla
  • Recién liberado de una wait condition — fue notificado, espera para retomar ejecución
  • Recién completó una operación signalC — señalizó y puede continuar

Condition Variables

Las condition variables son el mecanismo que tienen los procesos para esperar dentro del monitor a que se cumpla una condición. Una condition variable C:

  • No guarda ningún valor por sí misma
  • Tiene asociada una cola FIFO de procesos esperando
  • Consta de tres operaciones atómicas
OperaciónEfectoDiferencia con semáforo
waitC(cond) Agrega el proceso a la cola FIFO, lo bloquea, y libera el lock del monitor para que otros puedan entrar. Siempre bloquea (a diferencia de wait de semáforo que puede no bloquear)
signalC(cond) Desbloquea el proceso del tope de la cola. El proceso señalizador puede continuar o esperar, según la semántica. No tiene efecto si la cola está vacía (a diferencia de signal de semáforo que siempre tiene efecto)
empty(cond) Retorna true si la cola de la condition variable está vacía (nadie espera). No existe en semáforos

Semáforo vs Monitor: tabla comparativa

SemáforoMonitor (Condition Variable)
wait puede o NO bloquearwaitC siempre bloquea
signal siempre tiene efectosignalC no tiene efecto si la cola está vacía
signal desbloquea un proceso arbitrariosignalC desbloquea el proceso del tope de la cola (FIFO)
Proceso desbloqueado puede continuar inmediatamenteProceso desbloqueado debe esperar a que el señalizador deje el monitor

Condition Variables en Rust (Mutex + Condvar)

En Rust, los monitores se implementan con la combinación de Mutex<T> y Condvar. El patrón típico es envolver ambos en un Arc para compartirlos entre threads:

rustuse std::sync::{Arc, Condvar, Mutex};

// Patrón común: (Mutex con el estado, Condvar para señalizar)
let pair = Arc::new((Mutex::new(false), Condvar::new()));
let pair2 = pair.clone();

// Thread que ESPERA (consumidor)
std::thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut ready = lock.lock().unwrap();
    
    // IMPORTANTE: siempre en un WHILE, no en un if
    // (por los spurious wakeups — despertares espurios)
    while !*ready {
        ready = cvar.wait(ready).unwrap();
        // wait: libera el lock, espera, retoma el lock al despertar
    }
    println!("Condición cumplida!");
});

// Thread que SEÑALIZA (productor)
let (lock, cvar) = &*pair;
let mut ready = lock.lock().unwrap();
*ready = true;
cvar.notify_all();  // despertar a todos los que esperan
// cvar.notify_one(); // despertar solo a uno
⚠
Spurious Wakeups (despertares espurios): Un thread puede despertar de un wait sin que notify haya sido llamado. Esto es un comportamiento permitido por POSIX. Por eso la condición siempre debe verificarse en un while, nunca en un if. Si usás if, el código puede continuar aunque la condición no se cumpla.

Monitores en Java

Java implementa monitores nativamente con la keyword synchronized. Cada objeto Java tiene un lock (monitor) asociado:

java// Bloque synchronized: lock sobre 'this'
public void incrementar(int cantidad) {
    synchronized(this) {
        this.valor += cantidad;
    }
}

// Método synchronized: equivalente, lock sobre 'this'
public synchronized void incrementar(int cantidad) {
    this.valor += cantidad;
}

// Señalización: debe tener el monitor adquirido
// wait() libera el monitor y suspende el hilo
// notify() despierta alguno de los hilos que espera
// notifyAll() despierta todos los hilos que esperan
ℹ
Variables volatile en Java: Los hilos guardan copias de las variables compartidas en sus caches. La keyword volatile indica al compilador que el valor no debe cachearse — siempre se lee de la memoria principal. Garantiza visibilidad entre threads, pero no atomicidad: no reemplaza a synchronized para operaciones compuestas.

Problemas Clásicos con Monitores

Clásico
El Problema del Barbero Dormilón
▼

Una barbería tiene una sala de espera con sillas. Si la barbería está vacía, el barbero se pone a dormir. Si un cliente entra y el barbero está durmiendo, lo despierta. Si el barbero está atendiendo, el cliente se sienta en una silla y espera su turno.

Solución con semáforos: Un semáforo clientes (0) cuenta los clientes esperando. El barbero hace wait(clientes) (duerme si no hay clientes). Cada cliente hace signal(clientes) (despierta al barbero). Se agrega un mutex para proteger el conteo de sillas disponibles.

Clásico
El Problema de los Filósofos
▼

Cinco filósofos se sientan alrededor de una mesa. Cada uno tiene un plato de fideos y un palito chino a la izquierda de su plato. Para comer necesitan dos palitos: el de su izquierda y el de su derecha. Si cualquier filósofo toma un palito y el otro está ocupado, espera con el palito en la mano.

Problema: Si todos toman el palito izquierdo simultáneamente → deadlock. Nadie puede tomar el palito derecho.

Soluciones: (1) Solo puede comer si ambos palitos están disponibles (atómico). (2) Uno de los filósofos toma los palitos en orden inverso. (3) Árbitro (waiter) que controla el acceso. (4) Solución de Chandy/Misra con canales de mensajes.

Clásico
El Problema del Lector-Escritor
▼

Un estado se comparte entre varios procesos. Algunos necesitan actualizar el estado (escritores), otros solo leerlo (lectores). Las reglas son: múltiples lectores simultáneos están permitidos, pero un escritor requiere acceso exclusivo total.

Proceso 1Proceso 2¿Permitido?
EscribiendoEscribiendo❌ No permitido
LeyendoEscribiendo❌ No permitido
EscribiendoLeyendo❌ No permitido
LeyendoLeyendo✅ Permitido

Problema de starvation: Si continuamente llegan lectores, el escritor nunca puede acceder (Writer Starvation). Variante "Writer Preference": los escritores tienen prioridad. Variante "Fair": los procesos acceden en orden de llegada.

Clásico
El Problema de los Fumadores
▼

Para fumar se necesitan 3 ingredientes: tabaco, papel y fósforos. Hay 3 fumadores, cada uno con un ingrediente infinito de uno solo. Un agente pone aleatoriamente dos ingredientes en la mesa. El fumador que tiene el tercer ingrediente los toma, hace un cigarrillo, fuma, y el agente vuelve a poner dos ingredientes.

Este problema ilustra el problema de la signalización con semáforos cuando la señal debe despertar a un proceso específico (el que tiene el ingrediente complementario), pero los semáforos no proveen esta selectividad directamente.

10 · Formalismo

Redes de Petri

Formalismo matemático para modelar sistemas concurrentes. Permite razonar sobre propiedades como deadlock, alcanzabilidad y corrección de manera visual y rigurosa.

●
Lugar (p): representa un estado o condición
Transición (t): representa un evento o acción
Token (•): indica que un lugar está marcado / activo

Red Ordinaria de Petri

Definición Formal

PN = (T, P, A) donde T = transiciones, P = lugares, A ⊆ (T × P) ∪ (P × T) = arcos. Una transición t está habilitada si M(p) ≥ 1 para todo p ∈ I(t). Al disparar: consume tokens de entrada, produce tokens de salida.

Red General de Petri (con pesos)

Definición Extendida

PN = (T, P, A, W, M₀) donde W : A → ℕ es la función de peso. Una transición t está habilitada si M(p) ≥ W(p, t) para todo p ∈ I(t).

11 · Redes de Petri · Avanzado

Red General & Ejemplos Clásicos

Modelo Petri
Mutex (Exclusión Mutua)
▼

P2 inicializado con 1 token actúa como el lock. Solo uno de los procesos puede tener ese token y entrar a la sección crítica. P3 y P4 nunca tienen token simultáneamente → exclusión mutua garantizada.

Modelo Petri
Productor-Consumidor con Buffer Acotado (N)
▼

Se agrega lugar p₆ con N tokens que representa los espacios libres. La transición que produce necesita un token de p₆ (hay espacio). La que consume devuelve un token a p₆. El productor no puede superar N items.

Modelo Petri
Lector-Escritor con Arco Inhibidor
▼

Arco Inhibidor: Se representa con flecha de punta abierta. Una transición con arco inhibidor desde p solo está habilitada si p NO tiene tokens. T0 (lector) tiene arco inhibidor desde P4 (escritores activos). T2 (escritor) tiene arco inhibidor desde P1 (lectores activos).

12 · Pasaje de Mensajes

Channels & Mensajes

Modelo de comunicación basado en el pasaje de mensajes entre procesos. En lugar de memoria compartida, los procesos se comunican enviando datos a través de canales tipados.

💬
Filosofía Go adoptada por Rust: "Do not communicate by sharing memory; instead, share memory by communicating." En vez de compartir una variable con locks, usá un canal para enviar el dato.

Modelos de Comunicación

Comunicación Sincrónica

El emisor se bloquea hasta que el receptor recibe el mensaje. No hay buffer. Garantiza que el receptor procesó el mensaje antes de que el emisor continúe.

Comunicación Asincrónica

El emisor deposita el mensaje en un buffer y continúa sin esperar. El receptor lo lee cuando puede. Más flexible pero requiere manejo del buffer.

Direccionamiento Simétrico

El emisor nombra al receptor y el receptor nombra al emisor. Ambos se conocen.

Flujo Unidireccional

Los canales solo transportan datos en una dirección: de emisor a receptor. Para comunicación bidireccional se usan dos canales.

Canales en Rust — std::sync::mpsc

Rust provee canales MPSC (Multiple Producer, Single Consumer). El canal tiene dos extremos: Sender<T> (se puede clonar para múltiples productores) y Receiver<T> (un solo consumidor). Transfieren el ownership del elemento enviado — no hay copia.

rustuse std::sync::mpsc;
use std::thread;

// Crear el canal: tx = transmitter (sender), rx = receiver
let (tx, rx) = mpsc::channel();

// Múltiples productores: clonar el sender
let tx2 = tx.clone();

thread::spawn(move || {
    tx.send(String::from("Hola desde thread 1")).unwrap();
});

thread::spawn(move || {
    tx2.send(String::from("Hola desde thread 2")).unwrap();
});

// recv() bloquea hasta recibir. El canal cierra cuando todos los tx se dropean
for received in rx {  // rx implementa Iterator
    println!("Recibido: {}", received);
}

Canales en Unix: Pipes, FIFOs y Message Queues

Unix provee Pipes y FIFOs para conectar dos procesos independientes, orientados a bytes. Los FIFOs tienen representación en el filesystem. También existen Message Queues, orientadas a mensajes como unidades independientes.

Selective Input

Selective Input

Sintaxis que permite escuchar en varios canales de forma bloqueante y desbloquearse con el primero que recibe un mensaje. En Go se llama select. Fundamental para que un proceso pueda reaccionar a distintas fuentes de datos.

Filósofos con Canales — Solución de Chandy/Misra

✓
Idea clave: Los palitos circulan por mensajes entre filósofos. Cada palito puede estar sucio o limpio. Un filósofo entrega un palito sucio si le es solicitado, pero retiene uno limpio. Esto garantiza ausencia de deadlock y starvation sin memoria compartida.
13 · Modelo de Actores

Actores & Actix

Modelo de concurrencia desarrollado por Carl Hewitt en 1973. Popularizado por Erlang. Los actores encapsulan estado, comportamiento y tienen una dirección única.

📬
Mailbox (FIFO)
Cada actor tiene una cola de mensajes entrantes. Procesa uno a la vez, nunca hay acceso concurrente a su estado.
🔒
Estado Privado
No comparte memoria con otros actores. El estado solo cambia al procesar mensajes. Sin locks.
🌐
Distribuible
La dirección de un actor puede ser remota. El código no cambia entre local y remoto.
⚡
Livianos
Se pueden crear miles o millones de actores en lugar de threads del SO.

Formas de Enviar Mensajes (Actix)

MétodoComportamiento
Addr::do_send(M) Fire-and-forget. Ignora errores. No retorna resultado. El actor puede no existir.
Addr::try_send(M) Envío inmediato. Retorna SendError si el mailbox está lleno o el actor cerrado.
Addr::send(M) Retorna un Future con el resultado del handler. Permite .await para esperar la respuesta.
14 · Sistemas Distribuidos

Concurrencia Distribuida

En sistemas distribuidos, los procesos corren en máquinas separadas y se comunican por red. Los algoritmos clásicos de exclusión mutua y comunicación deben adaptarse a este modelo.

Exclusión Mutua Distribuida

Centralizado
  • Un proceso es el coordinador
  • Para entrar a la SC: enviar mensaje al coordinador
  • Si SC libre: coordinador responde OK
  • Si SC ocupada: coordinador encola y espera

⚠ Punto único de falla

Ricart-Agrawala
  • Enviar REQUEST con timestamp a todos
  • Recibir OK de todos para entrar
  • Timestamps desempatan conflictos
Token Ring
  • Procesos en anillo lógico
  • Solo el que tiene el token puede entrar
  • Al salir, pasa el token al siguiente

15 · Distribuido II

Algoritmos de Elección

Varios algoritmos distribuidos requieren un coordinador con un rol especial. Los algoritmos de elección permiten elegir uno automáticamente cuando el actual falla.

Bully (Garcia-Molina)

Cuando el coordinador no responde, P envía ELECTION a los de mayor ID. Si nadie responde, P es el coordinador. El mayor ID siempre gana. O(N²) mensajes en el peor caso.

Ring

Procesos en anillo lógico. El mensaje ELECTION recorre el anillo acumulando IDs. Al completar la vuelta, el mayor ID gana. O(N) mensajes.

Comparación
  • Bully: simple, re-elección automática
  • Ring: eficiente en anillos
  • Token Ring: equitativo, sin starvation
16 · Distribuido III

Transacciones Distribuidas

Una transacción es una secuencia de operaciones que debe ejecutarse de forma atómica en un sistema distribuido. Si cualquier parte falla, todo debe revertirse.

Propiedades ACID

A
Atomic
La transacción no puede ser dividida. Todo o nada.
C
Consistent
Cumple con todos los invariantes del sistema.
I
Isolated
Las transacciones concurrentes no interfieren entre sí.
D
Durable
Una vez commiteados los cambios, son permanentes.

Commit en Dos Fases (2PC)

⬡ Fase 1 — Prepare
1
El coordinador escribe prepare en su log y envía PREPARE a todos los participantes
2
Los participantes que pueden commitear responden READY y escriben en su log
3
Si alguno responde ABORT, la transacción se cancela
⬡ Fase 2 — Commit
1
Si todos respondieron READY: el coordinador escribe COMMIT y envía mensaje COMMIT
2
Los participantes escriben COMMIT en el log y responden FINISHED
3
El coordinador escribe END OF TRANSACTION
Técnica
Private Workspace, WAL, 2PL y Concurrencia Optimista
▼

Private Workspace: Al iniciar una transacción, el proceso recibe una copia de todos los archivos a los que tiene acceso. Trabaja con la copia. Al commitear, se persisten los cambios. Extremadamente costoso salvo por optimizaciones (copy-on-write).

Writeahead Log (WAL): Los archivos se modifican in place, pero se mantiene un log de los cambios. Al commitear, se escribe un registro commit en el log. Para abortar, se lee el log de atrás hacia adelante para hacer rollback.

Two-Phase Locking (2PL): Fase de expansión: tomar todos los locks. Fase de contracción: liberar todos (no tomar nuevos). Garantiza serializabilidad. Pueden ocurrir deadlocks.

Concurrencia Optimista: Modificar sin control, esperando que no haya conflictos. Al commitear, verificar si hubo conflictos. Si los hubo, abortar. Libre de deadlocks pero costoso bajo alta contención.

17 · Distribuido IV

Ambientes Distribuidos

Modelo formal para describir sistemas distribuidos: entidades, eventos, comportamiento colectivo y métricas de costo.

Entidad

Unidad de cómputo de un ambiente informático distribuido. Tiene acceso de lectura/escritura a memoria local (no compartida), capacidad de procesamiento local, comunicación via mensajes, y un reloj local.

Eventos Externos

La entidad es reactiva: solo responde a eventos externos. Los posibles eventos son:

  • Llegada de un mensaje — enviado por otra entidad
  • Activación del reloj — el reloj local dispara una alarma
  • Impulso espontáneo — el único evento que viene de fuera del sistema

Reglas y Comportamiento

Regla

Relación entre evento y estado: estado × evento → acción. Para cada posible evento y estado debe existir una única regla.

Comportamiento Homogéneo

Cuando todas las entidades del sistema tienen el mismo comportamiento. Todo comportamiento colectivo puede transformarse en homogéneo.

Costo y Complejidad

📨
Costo de Mensajes (M)
Cantidad de transmisiones y carga de trabajo por entidad y de transmisión.
⏱
Tiempo de Ejecución
Tiempo total del protocolo. Tiempo ideal: medido bajo delays unitarios y relojes sincronizados.
18 · Distribuido V

Deadlocks Distribuidos

En sistemas distribuidos, los deadlocks son más difíciles de detectar porque no existe una vista global del estado. Se necesitan algoritmos especiales para detectarlos y prevenirlos.

Algoritmo
Centralizado: Grafo de uso de recursos
▼

El proceso coordinador mantiene el grafo de uso de recursos. Los procesos envían mensajes al coordinador cuando obtienen/liberan un recurso, y el coordinador actualiza el grafo. Problema: los mensajes pueden llegar desordenados y generar falsos deadlocks. Solución: usar timestamps globales (algoritmo de Lamport) para ordenar los mensajes.

Algoritmo
Distribuido: Probe Messages
▼

Cuando un proceso debe esperar por un recurso, envía un probe message al proceso que lo tiene. El mensaje contiene: id del proceso bloqueado, id del emisor, id del destinatario. Al recibir el mensaje, el proceso lo reenvía a quienes tienen el recurso que necesita. Si el mensaje llega al proceso original, tenemos un ciclo en el grafo → deadlock detectado.

Prevención mediante Timestamps

Wait-Die

Si el proceso tiene timestamp menor (más viejo), espera. Si no (es más joven), aborta la transacción y vuelve a intentar.

Wound-Wait

Si el proceso tiene timestamp menor (más viejo), aborta la transacción del proceso que tiene el recurso (lo "hiere"). Si no, espera.

19 · Extra · GPU

CUDA & Programación GPU

Las GPUs tienen miles de núcleos simples optimizados para operaciones masivamente paralelas. CUDA es el modelo de programación de NVIDIA para aprovechar este poder de cómputo.

Warp

La ejecución se schedulea en warps de 32 hilos. Todos ejecutan el mismo código — comparten un único program counter. Cada hilo trabaja en un "lane" SIMD. La performance óptima se logra cuando la cantidad de threads por bloque es múltiplo de 32.

Jerarquía de Threads

G
Grid

El programa se divide en un grid de bloques independientes. Representa el total del trabajo a realizar.

B
Block of Threads

Cada bloque tiene N threads en el mismo SM (Streaming Multiprocessor). Acceden a Shared Memory compartida.

T
Thread

Unidad mínima. Se identifica con threadIdx.x dentro del bloque. Índice global: blockIdx.x * blockDim.x + threadIdx.x

Modelo de Memoria

TipoAlcanceVelocidad
RegistersPor thread (private)⚡ Mínima latencia posible
Shared MemoryPor bloque (todos los threads del bloque)⚡ Muy rápida
Global MemoryTodos los threads de la grid🐢 La más lenta

Loom

Herramienta para testear programas concurrentes. Ejecuta los tests muchas veces, permutando las posibles ejecuciones concurrentes según el modelo de memoria C11. Permite cubrir todos los posibles escenarios de ejecución sin explosión combinatoria.

mockall
